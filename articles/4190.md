# “那一天，Google L7级面试官把我的系统设计批得体无完肤”

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/S1iaf4GgGjExRCstE0oaRaBjicibibgE9Bn11QuUZK6QVmYWibCCEiajWhVGoYej9lhu4c0nnDtlw4kVxky4DNWczA2qc5OCekHqrAamwaKbTP540/640?wx_fmt=jpeg&from=appmsg)

在顶级互联网公司面试中，光靠背模式、画图和记住技术名词远远不够。

本文作者通过一次 Google L7 面试，深刻体会到系统设计的真正考验——不是“能画出漂亮架构图”，而是能否理解系统为何而生、如何应对极端故障，以及在复杂权衡中做出合理决策。

从 URL 缩短器的小题目，到 Netflix Chaos Monkey 的极端案例，本文带你透视高级工程师眼中的系统设计逻辑，理解“纸上建筑师”和真正实战高手之间的差距。

原文链接：https://medium.com/expocomputing/google-l7-system-design-interview-lessons-0b3834fded07

作者 | Cloud With Azeem  责编 | 苏宓  

出品 | CSDN（ID：CSDNnews）

我走进面试房间时，带着一种“胸有成竹”的淡定自信。因为我记住了所有蓝图，知道 MongoDB 和 Cassandra 的区别，能闭着眼睛画出内容分发网络（CDN）的架构，Netflix 的技术栈就像圣经一样刻在我的脑海里。

我准备好谈论微服务、分片和高可用性了。

然后，我见到了面试官——一位说话轻声细语的 Google L7 级工程师。过去十年间，这位面试官一直在维护那些每秒处理流量比大多数应用一年还多的系统。

他给我的题目看似简单：“设计一个 URL 缩短器。”

看到这个题目，我轻笑了一下。

这不过就是系统设计的“Hello World”入门必备题目。于是，我拿起马克笔，开始飞快地勾勒出系统的各种细节。这里放 API Gateway，那里放负载均衡器，一个 NoSQL 数据库存储映射，再加一个 Redis 缓存保证响应迅速。我甚至还在面试时候提到了 O(1) 查找和 Base62 编码。

到这里，我觉得自己的表现非常不错，感觉自己像个摇滚明星。

此时，他向前倾身，向我抛了一个问题：“这对一个只有几千用户的初创公司来说，确实不错。但如果写入量达到每秒 1000 万呢，你的架构会发生什么变化？特别是，你选的数据库在那个规模下如何处理预写式日志（WAL）竞争？”

因为这个问题，我的马克笔悬在半空，冷汗顺着背脊滑下。我懂模式，但不懂这些系统背后的物理逻辑。我意识到自己只是个“纸上建筑师”——一个能画图，但从没真正走过实际的地形的人。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06QuwjkSP1G6wEJHaJCLTONqlcQexqRgJcIICxofIOJs6B6tWBfibb7now/640?wx_fmt=png)

模式匹配的幻象

我们中的大多数人都会陷入模式陷阱。我们通过观察“大公司”的做法学习系统设计。看到 LinkedIn 放弃某些遗留系统，或者读到他们 Kafka 替代方案和新的流处理系统，就会立刻想：“好吧，LinkedIn 都这样做了，我也应该这样做。”

![](https://mmbiz.qpic.cn/mmbiz_png/S1iaf4GgGjEzD2YJjNRdhOby7cxhmqPr99vBmbQNwk74y3JkwtZy5aQHwePicFjFYoic8L2oTSNLr9AVEvOUKf6OJXRqdibJHwJeKxNGlDXxlV8/640?wx_fmt=png&from=appmsg)

但 L7 面试官并不关心你能否复制 LinkedIn。他关心的是你是否理解 LinkedIn 为什么必须这么做。

大厨与学徒的类比

把系统设计比作烹饪。学徒厨师按食谱做事（模式）。食谱说“加盐”，他们就加盐。但大厨懂化学原理。如果今天的番茄更酸，他会调整糖量。他不会盲目照搬食谱，而是根据食材调整。

在系统设计中，你的“食材”是约束条件：

*   延迟预算（有多快？）  
    
*   吞吐量要求（处理多少？）
    
*   数据一致性（有多准确？）
    

如果你无法告诉我为什么选择某个工具——比如为什么可能放弃 Terraform，改用更可编程或状态管理的方案——那么你不是在设计，你只是背诵。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06QDyP5HTUOXsPlJWd79yygiasaqXicpN7ibIfqiak5WFpaxE1mGxxpfmMjiaA/640?wx_fmt=png)

横向扩展并非万能解

那次面试中，最让我豁然开朗的时刻之一是关于讨论扩展性时。我一直说：“我们只要增加节点就行了。”横向扩展是初级到中级工程师常用的万能办法。

面试官打断我：“你每加一个节点，就增加一次网络跳转。每次网络跳转都有故障概率。你的 1000 个节点的协调开销什么时候会比单台高度优化的垂直机器更慢？”

案例研究：集群挤兑问题（Thundering Herd）

想象一下，一家世界闻名的面包店，每天 9 点免费发杯子蛋糕。如果只有一扇门，大家都会拥到门口（垂直瓶颈）。如果加十扇门（横向扩展），人群会分散。但如果十扇门都通向同一盘蛋糕呢？现在，十倍的人同时冲向单点，造成拥挤。

在 URL 缩短器中，如果某条链接病毒式传播（比如超级碗广告），那么你的每个“横向扩展”的服务器都会同时击打数据库的同一行或缓存的同一 key。这就是 Hot Key 问题。加服务器反而更糟，因为增加了同一瓶颈的并发请求。

解决方法不仅仅是“扩展”。你可能需要请求合并（多个相同请求等待一次数据库读取）或自适应缓存。

这种思考方式区分了高级工程师和普通工程师。

失败的架构：从混乱中推理

Google 的 L7 工程师没问我“这个是怎么工作的”，而是问“它怎么出问题的？”

他逼我思考极端情况：“你在 US-East 的主数据库着火了，US-West 的故障切换启动，但网络链路只开放 10% 带宽。用户会看到什么？”

我意识到我没考虑状态不一致。头脑中，“failover”是个魔法按钮，实际上如果处理不当，这是数据丢失的灾难。这就是单体 vs 微服务争论真正落地的地方。单体架构中这是一次大失败，微服务是成千上万的小失败，调试难度更大。

Netflix Chaos Monkey 案例

![](https://mmbiz.qpic.cn/mmbiz_png/S1iaf4GgGjEwVibJUSiar4QXic7aia5KwqUZq15KsQnODp3V2ibvEW3AU8KJn0toKBM88fA1FnmAxhpOlsI0vZoN5PBHMGReKrIGyCuzelAuwUq3I/640?wx_fmt=png&from=appmsg)

Netflix 就以这种做法而闻名。他们意识到，在分布式系统中，故障不是一种可能性，而是一种必然性。他们设计了 “Chaos Monkey”（混沌猴子），在生产环境中随机关闭服务器。

为什么？

为了迫使他们的工程师设计出具有自我修复能力的系统。

如果我设计一个每秒 1000 万写的 URL 缩短器，我必须假设：  

1.  缓存会宕机  
    
2.  磁盘会满
    
3.  短码生成器会产生重复
    

如果你的设计没有应对“黑天鹅”事件的方案，那它只是幻想。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06Q3SAo8dibFicWvibnB5u4gdHLgb2AbA6UJ8VImKnyW92hibqZefwpDbPYAQ/640?wx_fmt=png)

“无缓存”思维的力量

最让我震撼的时刻，是面试官挑战我对 Redis 的依赖：“不使用分布式缓存，重新设计。”

我愣住：“但…延迟会很糟！”

他反问：“真的会吗？如果 90% 的短链只被点击一次（数据的‘长尾’），你的缓存命中率只有 10%。这样每个请求都要先查缓存、再落到数据库，反而增加了延迟。你为 90% 的流量增加了 5ms 的开销，却毫无意义。”

这彻底颠覆了我的认知。以前学到的“Cache = 快”，实际有效延迟公式是：

有效延迟 = (命中率 × 缓存延迟) + ((1 — 命中率) × 数据库延迟)

当命中率低时，管理分布式缓存的开销（和陈旧数据风险）超过了收益。面试官想看到我是否有勇气说：“这里暂时不需要缓存。”

工程是权衡的艺术

两个小时后，我的白板布满划掉的框和重写的逻辑。但这是第一次，我感觉设计真实存在。它不是教材里的图，而是经过战火洗礼的计划。

我学到了，系统设计不是找“正确答案”。没有正确答案，只有权衡：  

*   一致性 vs 可用性：是保证用户看到精确数据，还是保证页面能加载，即便数据滞后几秒？  
    
*   延迟 vs 成本：是追求 <10ms 延迟，还是控制预算？
    
*   复杂度 vs 可维护性：是构建 50 个微服务的“完美”系统，还是一个三人团队能管理的可维护系统？
    

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06QGHX4EaTS3AknUHvnayNiaLnANlsxF3g1JR5q9HIdppdNd8c8uyuC05w/640?wx_fmt=png)

L7 面试该如何准备

要超越“死记硬背模式”的局限，你需要改变学习方法：  

1.  不只是画框，而要解释成本。每次加负载均衡或消息队列，问自己：成本是多少？延迟是多少？维护成本是多少？  
    
2.  从“裸机”开始。先设计只有一台服务器和一个数据库的系统，再根据数字证明需要增加复杂性。  
    
3.  学会数学。无需成为数学家，但要能估算 QPS、5 年存储需求、带宽需求。  
    
4.  研究真实的故障案例。读 AWS、Cloudflare、Slack 等公司的事故分析报告，看看他们“完美”设计在现实中如何失败。真正的学习就在那里。
    

停止像收集宝可梦卡片一样收集图案了，开始理解每条设计背后的“为什么”。在现实世界——以及高级面试中——白板不在乎你的图有多漂亮，它在乎你的逻辑。
