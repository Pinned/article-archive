# Agent 原生开发时代到来，Google Cloud Next 26 给开发者带来了什么

作者 | 唐小引

四月的拉斯维加斯，阳光热烈，比沙漠气候更火热的，是 Google Cloud Next 26，全球数万开发者齐聚。从机场领取 Badge 大排长龙开始，其火热气息就已经扑面而来。在 Day1 的 Keynote 现场，虽然正式开场时间是上午 9 点，但当我们在早上 7 点半抵达时，入口早已人头攒动，提前占位的队伍一眼望不到头。而到了第二天的 Developer Keynote，热情更是只增不减，我们甚至在门口足足排了三次队才成功挤入会场。

这种热度背后，是整个行业对 AI 进入 Agent 原生时代的高度关注。在会场的各个 Session 和展台交流中，我们发现大约 80% 的关键词都是 Agent。这不是简单的技术热点，而是一场深刻的产业变革正在发生。CSDN 高级副总裁、奇点智能研究院院长李建忠认为，Google Cloud 在这次大会上系统性地展示了其对 Agent 原生时代的战略思考与技术布局，从云计算范式的重构到软件开发模式的革新，从企业智能应用的转型到智能体操作系统的定义，每一个环节都在向开发者传递一个清晰的信号：Agent 已经成为下一代主流的应用范式。

云计算的底层逻辑变了，三层同步转换

Day1 Keynote 最核心的信息，是 Google Cloud 对云产业范式转换的系统化布局，云正在经历从 Cloud Native 到 Agent Native 的转变，而且是三层同步转换。

理解这个判断，先要理解 Google 的特殊位置。它是一家真正意义上的全栈 AI 公司，从算力层的 TPU（这次发布第八代，推理和训练都做了更系统的优化），到模型层的 Gemini（全尺寸、多模态、世界模型、边缘端侧全覆盖），再到 Google Cloud，以及终端 Android 操作系统。从基础设施到企业应用的全栈视角，决定了它和纯粹模型厂商看 Agent 时代的方式截然不同。

三层的变化，对开发者来说每一层都很具体。

*   IaaS 层：从硬件租赁到 Token 工厂
    

传统的 IaaS 层提供的是硬件租赁和虚拟化资源，但在 AI 时代，尤其是有了推理对 Token 生产的巨大需求之后，云基础设施的本质发生了变化。Google Cloud 推出的第八代 TPU，针对推理和训练都做了更系统全面的优化，特别是推理侧全面回应了 Agent 对计算的要求，将传统的计算存储资源转变为 Token 的生产工厂，你用 Agent 写的每一行代码，最终都落在 Token 生产工厂的产出。这是 IaaS 层的一次巨大转换，从提供算力到提供智能生产能力。

*   PaaS 层： Enterprise Agent Platform 的诞生
    

传统的 PaaS 层是针对云原生时代的应用，提供管理和运行时的各种支持。而在 Agent 时代，云里面运行的应该是 Agent，因此需要一个 Agent Native 的运行环境。Google Cloud 推出的 Enterprise Agent Platform 正是这样一个支撑新 Agent 的 PaaS 层，包括 Agent 的整个运行、监管、安全治理、编排以及优化等功能。

特别需要指出的是，这个平台围绕 Agent 在企业应用中的核心痛点：“数据”和“安全”做了很多工作。企业的数据往往散落在各处，而 Google Cloud Data 方案能够把结构化和非结构化数据以 Agent 友好的方式整合在一起，提供给 Agent 作为全量的上下文数据。同时，在安全权限方面，Agentic Defense 围绕 Agent 的安全特点做了很多创新, 确保 Agent 既能发挥智能创意，又能遵循企业的安全规范。

*   SaaS 层： Agent as a Service 的演进
    

传统的 SaaS 层提供的是各种应用服务，而在 Agent 时代，这些应用服务正在转变为 Agent as a Service。无论是财务应用、客户关系管理还是销售管理，都在逐步 Agent 化。Agent 将成为用户的第一接口，改变人与软件应用交互的方式。

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/iaM6L8b5B1Gn82hBlxrk6CFxq1ZFJAVHCJ1MicMxWfdt0DweSMdSdGeKzuaPibTm0sDApshXib3ITOD5DbqWfkAVibNYSxPkNcb4jBZSAE23Iyhg/640?wx_fmt=jpeg&from=appmsg)

Google Cloud 在这三层上的思考非常系统化，尤其是面向企业级市场，展示了大量的企业应用案例。从底层 IaaS 到 PaaS 再到 SaaS 层的三层转换，Google Cloud 对云产业 Agent 化的推进有着全栈的视角、方法论以及系统工具。

更重要的是，Google Cloud 并没有把 Agent 看作一个简单的全新应用形式。Agent 的范式转换，很大程度上是因为传统软件需要用户去使用，流程和知识需要人类员工去执行。而新的 Agent 作为操作系统，本质上是一个虚拟的数字劳动力，是替代传统劳动力的一个新范式。

这也带来了人与 Agent 协作方式的变化。去年大家还在讲 Human in the loop，思考人如何介入 Agent 的运行过程，而现在正在发生的范式转换是 Human out of the loop。人退到 Agent 闭环之外做任务目标的设定，做行为监控，做效能的优化。人在闭环外不是彻底不做事了，而是变成 Agent 团队的指挥者、目标的定义者以及结果的评估者，干更高维度的活。

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/iaM6L8b5B1GnhAIpuicVdWwvWOmqLexDqAKTfib6MyRWuVGdsZibsk9LJ1wVIFSoNneU8xQZ0T9ibyCAnsmiay7lQ4tRNf4ZqHDYVY2ULTJoGWuibE/640?wx_fmt=jpeg&from=appmsg)

对软件开发者而言，这个变化意味着所需要培养的能力，除了要搭建 Agent 这一新的应用形态，还要从写被人类使用的传统软件，转向设计出能被 Agent 正确理解和使用的系统。

开发者的跃迁：告别“手搓”，构建 Agent 工程化闭环

在 Day2 的 Developer Keynote 上，焦点完全回归到了开发者身上。Google 这家公司在开发者生态上一直极具品味，其产品深得开发者喜爱。如今的核心任务，是帮助开发者完成从传统云原生工程向 Agent 工程的跨越。

李建忠指出：当前的 Agent 开发，业界一个共识是单纯的模型是不够的，模型同时必须要有系统配套。这也是当前业界对于驾驭工程（Harness Engineering）热捧的原因。Agent 驾驭工程的三个核心要素闭环：知道、行动、反馈。Google Cloud 通过一系列 Agent 开发和管理套件系统化地将其变为落地能力。

![](https://mmbiz.qpic.cn/mmbiz_png/iaM6L8b5B1GnZq55G8TAuPWWjLA9TicrryrlAepaQGv8j01wkhTs8pxcW6ACtIlRrIlO0Xv9CHpVD3LWUaUDGR1U7ZoYDfnvribbCea6laoKic4/640?wx_fmt=png&from=appmsg)

*   “知道”：Memory Bank 与上下文管理
    

Agent 驾驭工程的第一个要素是“知道”。Google Cloud 推出了 Memory Bank，用于管理整个企业里各种上下文的记忆。这包括 Skills、结构化数据、非结构化数据等各种形式的知识。Google Cloud 对企业级方案有着非常深刻的理解，能够将散落在企业各处的数据整合成 Agent 的外脑知识库。

*   “行动”：工具编排与 MCP 支持
    

Agent 的第二个要素是行动能力，这需要有一套工具配套。Google Cloud 的 Agent Runtime 层提供了对 MCP（Model Context Protocol）、Agent 注册表等的支持，使得 Agent 能够合理编排在整个任务路径上需要的各种工具以及权限配备。

从传统 API 向 Agent 友好的接口转换是一个重要趋势。传统的 REST API 是为传统应用和服务设计的，而 Agent 需要的是更加语义化、更加智能的接口，比如 MCP 或者 CLI 等。这种转换不仅仅是接口形式的改变，更是交互模式的革新。

*   “反馈”：可观测性与闭环优化
    

Agent 驾驭工程的第三个要点是反馈的闭环建立。Google Cloud 推出了一系列管理工具，包括可观测性、调试、追踪等功能，帮助开发者优化和观测 Agent 在整个闭环过程中的每一个路径。

评估框架在 Agent 开发中变得尤为重要。它不应该是事后才补上的环节，而应该从项目一开始就成为核心部分。开发者需要知道如何衡量项目是否成功，与过去的方案相比表现如何，而评估就是整个过程里的指挥器。通过评估框架，开发者可以形成反馈闭环，不断实验和迭代，观察 Tool calling 的效果、MCP Server 的支持是否有帮助，以及这些方法对 Token 消耗、成本和延迟的影响。

*   从手搓到系统化开发
    

Google Cloud 的这套组合产品，很好地解决了开发者在企业里开发复杂 Agent 应用时的关键问题。用 Google Cloud 的角度来讲，就是 Agent 的这套运行体系，把 Agent Harness 工程的“知道”、“行动”、“反馈”三个要素闭环串起来。

目前大家都在手搓各种 Agent 工程，但有了这些套件，企业团队协作就有了共同的接口基础。随着模型能力的不断演进，Google Cloud 面向开发者提供的工具支撑和配套的基础设施，相信会加速 Agent 大规模开发时代的到来。

*   A2UI，生成式 UI 的探索
    

Google Cloud 的 Agent 开发平台有很多非常棒的工具支撑，支持开发人员从之前手工去做 Agent 的每一个点，到一个有系统框架和工具支撑，甚至部分自动化、自主地去完成整个 Agent 工程。这次大会上还有一些亮点值得关注，比如 A2UI（Agent-to-User Interface）。这项技术很好地解决了交互层的 Agent 化，用 Agent 生成 UI 的方式去适配各种不同的平台，支持用自然语言描述去实现 UI 的自动化生成。为此，我们在逛展区时，和 A2UI 展位的专家特别聊了聊。

A2UI 的核心思路是用 Agent 动态生成 UI 界面。“A2UI 本质上是一个通讯协议，它为你提供的是一套 UI 规范。”你给 Agent 一个组件库，告诉它现在有 18 种组件可用，按钮、复选框、数字选择器、滑动条、日期时间输入框等等，Agent 根据用户的自然语言输入，决定返回什么样的 UI。返回的不是 HTML 或样式代码，而是一串 JSON，纯粹描述“界面应该长什么样”，然后由前端负责渲染，React、Angular、Vue、Flutter 都有对应的渲染库。

来自 Google Cloud 的专家强调“A2UI 完全开放。任何大模型、任何 Agent，不管是 Anthropic 的还是 OpenAI 的，都能支持这套标准。它只负责传递一个核心信息，智能体决定要给用户展示什么样的界面。”跨平台、开放兼容、与具体模型无关，这让 A2UI 的边界比一个普通产品功能要宽得多。专家演示了一个例子，让 Agent 列出 Google Cloud 上运行的所有服务，以前返回一大段纯文字，现在返回卡片式界面，直接点“查看详情”或“重启服务”，触发真实的后台操作，整个交互路径大幅缩短。从开发者的视角看，A2UI 和 MCP 是同一类东西，MCP 解决的是 Agent 的工具调用层，A2UI 解决的是 Agent 的交互呈现层。Agent 的协议栈，正在一块一块地被搭起来，而每一块都是开放的。

对话 Google Cloud：开发者生态的深度洞察

在大会期间，我们有幸遇到了两位 Google Cloud 开发者生态的关键人物：开发者市场高级总监 Ricky Robinett 和开发者关系总监 Karl Weinmeister，与他们的交流让我们对 Agent 时代的开发者生态有了更深刻的理解。

*   与 Ricky Robinett 对话：编程民主化与开发者定义
    

Ricky Robinett 对 Agent 时代的开发者生态持乐观态度。他认为，这场变革非常像当年向 Serverless 架构的过渡期。在那个阶段，开发者逐渐意识到需要一套全新的方式来观测系统。而现在开发 Agent 应用时，传统的 console.log 调试方式已经不再适用，开发者需要一个专门为此打造的底层平台，可观测性成为核心需求。

Ricky 特别喜欢用 Serverless 来类比 Agent 的构建方式。这些 Agent 就像是随时待命的数字员工，一旦被唤醒就会去处理任务，完事后又回去休眠。因此，开发者需要习惯这种全新的开发范式，同时也需要一个为此准备就绪的平台。

关于开发者技能的演变，Ricky 的观点令人振奋。他认为，每个人内心深处都有创造的冲动。过去程序员圈子里有个老段子：每次别人一听说你是程序员，就会说“我有个做 App 的绝妙点子，你能帮我写出来吗？”而现在，他们自己就能把 App 做出来了。

最让 Ricky 着迷的一点是，过去开发者需要苦哈哈地学习各种工具和编程语言，但现在的工具已经简单到几乎完全顺着直觉在运作。真正的转变在于如何像开发者一样去思考，如何去构思要构建的东西。这意味着形形色色的人都能加入这场盛宴，包括那些视障人士，他们过去必须艰难地摸索代码语法，但现在只要使用自然语言就可以了。

关于程序员数量的未来，Ricky 是个乐观主义者。他认为代码终究是计算机的语言。能够引导计算机去完成任务的能力将依然是一项极其宝贵的技能。如果我们把“开发者”定义为“那些与计算机对话、并引导计算机做事的人”，那么未来的机会只会越来越多。产品经理、运营、设计师，越来越多原本不写代码的人将能用自然语言构建真实的应用。“真正的转变在于，如何像开发者一样去思考，如何构思要构建的东西。”

我们提到中国有很多优秀的创业者在寻求全球创新，Ricky 特别分享了他到北京交流的经历。他表示，无论走到哪里，都能遇到无数聪明绝顶、才华横溢、满脑子奇思妙想的人。如果这个新时代赋予了更多人创造的能力，那就意味着创造者将会像雨后春笋般在全球各个角落涌现。Google 的使命就是尽己所能，赋能世界各地的每一位开发者，通过遍布全球的 Google 开发者节（DevFest）和 Google 开发者社区（GDG），支持全球开发者的创新。

*   与 Karl Weinmeister 对话： Harness Engineering 的未来
    

Karl Weinmeister 来自 Developer Relations 团队，他的工作是和全球开发者打交道。在谈到 Agent 开发时，Karl 特别强调了 Google Cloud 在 Next 上发布的 Agent 平台的重要性。这个平台把很多关键环节整合到了一起，不只是用 ADK（Agent Development Kit）来构建 Agent，还包括注册、安全、评估等能力，全部放在一个地方，让开发者更容易看到 Agent 从开发到落地的完整生命周期。

Karl 认为评估非常重要，应该从项目一开始就成为核心部分。现在 Google Cloud 把评估、记忆等框架都整合到了同一个平台里，让 Agent 开发者更容易上手。通过评估框架，开发者可以形成反馈闭环，尝试不同的 Prompt，使用 GEPA（Generative Evolutionary Prompt Adaptation）等新技术来迭代 Prompt，找到更好的结果。

关于 Skill 的概念，Karl 更倾向于把它看成是对 Agent 的一种训练，而不是某种替代品。一个 Agent 本身已经从训练数据里获得了一些基础知识，而 Skill 的作用是把它往正确的方向再推一把，给它一张路线图。Skill 的价值在于让 Agent 更高效，是一件聪明且值得做的事。

在谈到编程语言的未来时，Karl 认为编程语言仍然重要，因为不同语言在速度、类型安全、并发能力等方面还是有明显差异的。不过，一些过去学习门槛比较高的语言，现在没那么难进入了，因为 Agent 可以帮助开发者把能力延展到新的领域。开发者未必需要像以前那样过度担心语法细节，但仍然必须非常理解底层的核心概念，比如架构、模式、算法效率等。

关于传统软件的重构，Karl 提出了一个重要观点：传统的 REST API 并不能简单地一一对应转换成 MCP 工具。从 Agentic System 的角度来思考，交互模式是不一样的。开发者可能会把原来分散的多个 API 整合成一个 MCP 工具，需要聪明地设计这件事。另一方面，在应用结构设计上也可能会做出调整，比如可观测性会变得非常关键，让系统具备自愈能力，能够自己去看日志并做出决策，而不是中间还需要有人介入。

关于 Harness Engineering，Karl 认为 Harness 和模型都非常重要。模型本身必须足够强，提供足够大的上下文窗口和足够快的响应速度。但与此同时，Harness 也会继续重要，因为它关系到如何更聪明地使用模型，比如缓存响应、理解哪些内容应该进入 Memory 等。这里面有很多算法需要调优，而我们现在其实还在摸索。不同的 Harness 都有各自的优点和缺点，也会和个人风格相匹配，至少现在这里面还有很大的改进空间。

在谈到未来的开发者生态时，Karl 提出了几个值得考虑的方向。首先，开发者需要把自己的能力以 Skill 的形式提供出来，MCP Server 也很重要。文档最好能提供 Markdown 版本，这样能很方便地被 Agent 工具直接消费。另外，在 Agent 的帮助下，人们生产内容的速度会比过去快得多，这带来的结果是更需要真正的专业性。内容创作者要提供 Gemini 本身还不能直接提供的独特洞察，提供新的信息或某种真正属于人的独特视角。Karl 认为，个性这个维度会变得前所未有地重要。

带回来的一些判断

三天会程结束，我们认为有几个判断值得和国内开发者分享。

首先，李建忠认为，云计算产业正在面临 Agentic Cloud 的全面重构，传统的三层 IaaS/PaaS/SaaS 将被彻底改写。整个云计算产业将迎来剧变，这里面会带来很多创新的机会。 

随之而来的是，大规模 Agent 开发的时代正在到来。李建忠明确指出，Agent 工程正在告别手工，Memory、Runtime、Evaluation、可观测性等等，这些以前需要每个团队自己手搓的基础设施，现在有了系统化的工具支撑。评估先行，而不是事后补。这是 Karl 最实用的一条建议，也是我们观察到最多团队忽略的一件事。

开发者的定义在扩大。Ricky 说的那句话值得被记住，「引导计算机完成任务的能力，将依然是极其宝贵的技能，未来的机会只会越来越多。」跨界能力，在 Agent 时代是真正的超能力。
