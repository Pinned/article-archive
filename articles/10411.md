# IDE 正在消亡？

![](https://mmbiz.qpic.cn/mmbiz_gif/Pn4Sm0RsAujqPrtpN5QqqOuSG0AdUUx8wRg3vvr7aQJicdC0cHDBqZrJBZajQUXx97BwNaAE6L6x7OGlsTIpWjw/640?wx_fmt=gif)

自从 AI 辅助编码工具和各类 Agent 兴起以来，“IDE 已死”的说法层出不穷，仿佛传统代码编辑器已经毫无存在价值。

然而，Addy Osmani——这位在 Google 工作近 25 年、曾在 Chrome 团队主导 DevTools、Lighthouse 和 Core Web Vitals 的资深工程师，结合自己日常使用的多种 AI 工具给出了更细致的观察。

他表示，IDE 的确正在失去过去作为开发“重心”的独占地位，开发者的工作流程正向多 Agent 协作、调度和审查界面迁移，但这并不意味着它消失了。

代码编辑器仍然存在，只是对于越来越多的开发者而言，它不再是编程时打开的第一个界面而已。

来源：https://addyo.substack.com/p/death-of-the-ide

作者 | Addy Osmani        编译 | 苏宓  

出品 | CSDN（ID：CSDNnews）

开发者的工作“重心”正在发生变化。注意，不是消失，而是转移。

过去，我们习惯在一个代码编辑器窗口里一行一行地改代码。现在，越来越多的时间被花在“看着 Agent 干活”上，因为这些 Agent 会自己做规划、批量改文件、跑测试，然后把改动整理好，等人类来审核。

传统意义上的 IDE，可能不再是软件开发的绝对核心，要么被边缘化，要么彻底进化成另一种形态。

看看很多开发者（包括我）已经在日常使用的这些工具：Conductor、Claude Code Web、GitHub Copilot Agent、Jules、Vibe KanBan，甚至 cmux，你会发现一个共同趋势——真正的“主界面”正在变成调度和控制这些 AI Agent 的那一层，而代码编辑器，反而退居成底层的一种“工具组件”。

![](https://mmbiz.qpic.cn/mmbiz_jpg/S1iaf4GgGjEzjaQYRqf45Pnx0N5NywaMgiaGHYESXV6pZMkPbrZCN4sO0nyWhJuRBO85XdCSwZ9aY2tWURd3icDkzWcuhVRcYpoKD5PjY2dibvU/640?wx_fmt=jpeg&from=appmsg)

Cursor 最近发布了 Glass——一个全新的界面，其设计宗旨是让“与智能代理协作”这件事变得更清晰、更直观，而且始终在你的掌控之中。在这个界面设计里，管理 Agent 本身成了核心体验，而传统的代码编辑器，则变成你在需要深入细节时才会打开的工具。

这个工具发布之后，开发者的反馈也来得很快：

现在的 Cursor，更像一个“代理调度中枢”（Agent Orchestrator），而不是传统意义上的 IDE。并行管理多个 Agent，也变得轻松多了。

![](https://mmbiz.qpic.cn/mmbiz_png/S1iaf4GgGjEye82uqTJgYOsQJmNrj6J2sLQ3N3RHFNeonAUsr1ib1EciaeK1hWegCvDZ9N1kORGDrM4CUYU7sa2NMvadhwyIRHlq1fwwJhFpYs/640?wx_fmt=png&from=appmsg)

但 Glass 只是更大趋势中的一个缩影。像 cmux 这类终端界面也在说明一件事：我们熟悉的那些操作界面，正在被重新塑形，更好地服务于“多 Agent 协作”这种新的工作方式。

![](https://mmbiz.qpic.cn/mmbiz_jpg/S1iaf4GgGjEzbHCfbmJrqfibFELOuq8oHuhDne5CmiaWqGM3UksqDmKLsiaVOpgfejictMeaOEVic0Ul1ibuZQVjgyURZSv7f9XD3UoicdT5EnkAMiag/640?wx_fmt=webp&from=appmsg)

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06QuwjkSP1G6wEJHaJCLTONqlcQexqRgJcIICxofIOJs6B6tWBfibb7now/640?wx_fmt=png)

从“改文件”，到“控流程”

以前，IDE 都是围绕一套固定流程优化的：打开文件 → 改代码 → 编译 → 调试 → 再来一遍。现在业界经常有人说“IDE 已死”，是因为一旦 Agent 能自动干完大部分事，这套循环就不再是决定效率高低的关键了。

新的工作节奏，更像这样：明确目标 → 交给 Agent → 观察执行 → 审查改动 → 合并结果。它和“带聊天框的自动补全”最大的区别在于：Agent 不仅能用工具自主完成任务，而且配套的界面，也在围绕“如何把这种自主能力管住、用好”来设计。

这种变化，其实已经在一批高频工具中落地了。比如 Claude Code（Web 或桌面版）和 Codex，可以让开发者把结构清晰的任务直接交给运行在云端隔离环境里的 Agent 来完成，进度在浏览器里一目了然——不需要终端，也不用本地折腾环境。

GitHub Copilot 的 Agents 更进一步：它能自己规划并完成跨多个文件的修改，自动建分支、跑测试，最后把结果整理成一个 PR 交给你审。开发者的角色，从“手把手指挥每一步”，变成“看结果、提反馈、再迭代”。

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/S1iaf4GgGjEzM358qyU3jqWew27ia3QDePjscrNpoYoicksVCmlyxPXxF2iabCWiayM6XFgTzwjKibOyCGyG3hXAWZA7IeENnXc5wMEE1miaFYCHOI/640?wx_fmt=webp&from=appmsg)

Conductor 则走了另一条路：它是一个桌面应用程序，可以同时跑多个 Claude Code Agent，每个在独立工作空间里执行任务，你可以实时看到它们各自的进展。

而 Google 的 Jules 更像“异步外包”：你把任务分派出去，它在后台慢慢跑，完成后你再回来验收结果。

这些工具的核心思路都一样：干活的主体是 Agent，不是文件本身。真正值得优化的界面，是帮你分配任务、跟踪进度、审核结果的界面，而不是让你把代码敲得更快的编辑器本身。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06QDyP5HTUOXsPlJWd79yygiasaqXicpN7ibIfqiak5WFpaxE1mGxxpfmMjiaA/640?wx_fmt=png)

正在成型的“代理调度层”

只有当你仔细观察这些工具中趋同的界面模式时，“IDE 被替代”的说法才显得有说服力。

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/S1iaf4GgGjEzdGQd0Wz1XoUMGQNjD1hvlMhZFQYZ0zwLaiaD4uxMfsic336z5FJ5aLRFw4EmXficnx2I4UxNEUcwp2RwHPPYgq4kPNNicZgojZ0M/640?wx_fmt=other&from=appmsg)

工作隔离被当成了一项基础能力。并行运行的 Agent 之间必须避免互相干扰。在这一点上，几乎所有认真做这类工具的团队，都选择了 git worktree 或类似机制作为解决方案。Conductor 为每个 Agent 会话分配独立工作空间；Vibe Kanban 在其看板驱动的 Agent 工作流中也采用同样方式。这个模式几乎普遍存在，因为问题是真实存在的：没有隔离，多个 Agent 同时工作只会带来混乱。

以计划和任务状态为核心界面。像 Vibe Kanban 这样的工具，将传统的“标签页和文件”顶层思维模型替换为“任务与状态”。你创建任务卡片（比如一个落地页、后端服务或邮件集成）时，把它们分配给具体 Agent 和模型，然后像管理轻量级项目看板一样管理整个工作——唯一不同的是，“团队”是自动运作的。这种界面其实就是项目管理板，只是 Agent 在执行实现部分。

后台 Agent 与异步优先设计。一些最有意思的工具，根本不要求你在执行过程中全程盯着它们。Cursor、GitHub Copilot 和 Antigravity 都支持后台运行的 Agent——你定义目标，然后可以离开，完成后再回来查看结果就行。Jules 也是类似方式：分派任务，完成后再看差异。这里隐含的理念是：你的注意力太宝贵，不值得用来看进度条。这和 IDE 的实时同步反馈循环截然不同。

并行 Agent 的注意力管理。当多个 Agent 同时运行时，真正的瓶颈是——你如何知道现在该关注哪个 Agent。这就是为什么 Conductor 会展示所有会话的实时进度，而 cmux 在终端面板中引入了通知圈和未读徽章。“Agent 需要注意”正成为开发环境中的一级事件——需要你进行路由和分流，而不仅仅是看一眼。

嵌入软件生命周期的 Agent。GitHub 的 Copilot 编码代理是异步的，由控制层保护，并由 GitHub Actions 提供支持。它不仅跟代码的写作相关，更与实际发布流程紧密挂钩（issues → PRs → CI → merge）。

这些工具均未宣称 IDE 已过时——其中许多仍与 IDE 兼容互通。但上述反复出现的模式（并行工作区、基于差异的审阅、任务状态管理、后台执行、生命周期集成），正是“IDE 已死”论者所说的“开发工作重心转移”的核心体现。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06Q0QtgZAx8xsoTReptvArfwbn9MvHGVfV98Qkl5PMRS6MCt2ljwLJwoQ/640?wx_fmt=png)

为什么开发者仍然离不开 IDE？

对“IDE 已死”最有力的反驳是：IDE 仍然把几个真正复杂的问题压缩在一个高保真反馈循环里——精准导航、本地推理、交互式调试，以及通过直接操作系统来理解它的能力。

即便是最前沿的 Agent 调度工具，也保留了手动编辑的余地。例如，在工具里直接审查 diff、对改动发表评论，然后再在编辑器里打开结果进行手动调整。这本身就是对一个事实的承认：人工干预仍然是工作流程的一部分。

Agent 工具本身也暴露了它的局限性。对于大型仓库的多文件重构，仍然是软件工程 Agent 最难应对的挑战之一。这正是交互式代码导航和人类判断依然最有价值的场景——你需要在脑中构建系统模型，而 Agent 单凭上下文无法完整重建。

让开发者仍然依赖 IDE 进行深度检查的原因，是 Agent 往往“差一点就对”。当结果 90% 正确但存在不易察觉的 Bug 时，发现问题的成本往往比自己写一遍还高。

对于高风险改动，IDE 仍然是进行这种深度、精确检查的最佳工具。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06Q3SAo8dibFicWvibnB5u4gdHLgb2AbA6UJ8VImKnyW92hibqZefwpDbPYAQ/640?wx_fmt=png)

新的成本：审查疲劳与治理负担

如果开发变成“并行运行多个 Agent”，那么工作流继承的问题就不再像文本编辑，而更像分布式系统管理——可观察性、权限管理、隔离以及治理。

Agent 工作流颠倒了劳动方式。你不再是写代码，而是在审查代码。乍一听像是效率提升，但当你一天结束时盯着十二个并行 Agent 的十二个 diff 时，审查疲劳的感觉就非常真实。这也是为什么这一领域最用心的工具，更强调注意力分流、结构化计划和以审查为先的门槛，而不是默认追求完全自主。

随着 Agent 获得更多工具、代码库和外部系统的访问权限，安全面也随之扩大。Agent 可以浏览网页、查询数据库、写入文件系统、触发部署，因此它们“被允许做什么”变得和“能做什么”一样重要。

在可观察性和控制方面，IDE 集成的 Agent 模式已经推动了明确的工具日志和审批门槛。一旦 Agent 异步运行并触碰 CI 流程，治理问题就不再可选。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06Q0jqAkXB2hEohZkOqYzXVsmJtnNRQncYh54ZQOpcC5ZGSicYFgtBZP4w/640?wx_fmt=png)

未来存留下来的会是 IDE？还是控制层？亦或者两者？

从整体趋势来看，“IDE 消亡” 这一说法在开发重心转移这个大方向上是对的，但从字面预测来看并不准确。

![](https://mmbiz.qpic.cn/mmbiz_jpg/S1iaf4GgGjEzM4uu2EpL7XUw7sdPNiaUPcRovePJOlnxFn1icibDNly29fKSMRITEcpGtUrg1bMg1NTVTibolfo1HJ9FwhrSmHLYS8ciciaarfibRP8/640?wx_fmt=other&from=appmsg)

这一观点想要表达的是：IDE 不再是主要工作区，而是成为若干辅助手段之一——用于有针对性的检查、调试和最终编辑——而规划、调度、审查和 Agent 管理则转移到仪表盘、Issue 跟踪、可观察终端和云端控制层。

“更大的 IDE” 这一表述同样站得住脚。新一代 IDE 是一套完整系统，提供多 Agent 调度、隔离工作空间、权限与审计日志、以 diff 为核心的审查、可靠工具连接以及注意力分流的系统。代码编辑器仍然存在，只是不再是开发工作的入口。

IDE 并未消亡，只是被去中心化了。开发工作正在向外扩展——进入那些由人定义目标、分派给并行 Agent 运行时、更多时间用于监督、审查和治理的调度界面，而不只是手动敲代码。  

在保障代码正确性、理解复杂逻辑，以及处理智能代理仍难以胜任的难题时，IDE 依然至关重要。但它已不再是编程发生的唯一场所；并且对越来越多的开发者而言，它也不再是打开的第一个界面。
