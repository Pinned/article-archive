### **竞争的伪命题被戳破：如何管理成千上万个 Agent**

在过去的长达两年多的时间里，整个科技界和企业界都在围绕一个极其初级的问题打转：“我们能不能构建一个好用的 AI 助手？”

无数的企业在做 PoC（概念验证），在尝试用大语言模型去写一封邮件、生成一段摘要，或者解答一个简单的客服问题。但这种“孤立的实验”正在暴露出巨大的局限性——拼凑零散的芯片、调用孤立的模型、耗费数周的人工去硬编码编排逻辑，根本无法创造实质性的商业价值。

Google 兼 Alphabet 首席执行官 **Sundar Pichai** 在大会开场时，一针见血指出了当前企业 AI 的核心痛点：“大家讨论的问题，已经从「我们能构建一个智能体吗？」变成了「我们该如何管理成千上万个智能体？」 ”

![](https://mmbiz.qpic.cn/sz_mmbiz_png/iaM6L8b5B1GmlaQE4BlicpwH0HCUzmlSISZQbgP7GLeloTNzpq7WjKk1FjjRmKq9dJOof3SkmSIFNiciayS7Zout4zIbAG8RJshCnW8TyTiaSAgk/640?wx_fmt=png&from=appmsg)

这句话点到了企业上 Agent 之后真正会遇到的问题。

AI 一旦不只是回答问题，而是开始自己规划、自己调用工具、自己跨系统执行任务，问题就不再只是模型够不够聪明，而是这些东西怎么被管理、怎么被约束、怎么在企业里安全协同。财务部门可以随手做一个拉营收数据的智能体，市场部门也可以让智能体自动发送带定价策略的邮件。**如果上面没有统一系统，这套东西越灵活，风险反而越大。**

在 Google 内部，这种转变已经发生。Pichai 透露，**Google 现在接近 75% 的新代码由 AI 生成，再由工程师审核通过。**更重要的是，他们已经开始用多个智能体协同去重构开发流程。在一次复杂代码迁移里，负责规划、统筹、编码的不同智能体一起工作，把速度拉高了 6 倍。

当模型能力不再是稀缺资源，如何让成百上千个 Agent 在企业内部有序、安全、可预测地协同工作，才是真正的深水区。Google 看到了这一点，并决定以此作为其云业务下一阶段的最高战略纲领。

### **Unified Stack：不卖零散工具，卖“企业智能体操作系统”**

既然问题已经变成了系统级难题，解法自然不能是售卖单点工具。

Google Cloud 首席执行官 **Thomas Kurian** 在舞台上掷地有声地抛出了 Google 的解题思路：Unified Stack（统一技术栈）。

在过去的科技叙事中，云厂商习惯于各讲各的故事：硬件部门大谈芯片算力，AI 部门炫耀模型参数，数据部门推销数据库，安全部门强调防火墙。但 Kurian 此次勾勒的，是一张浑然一体的**“智能体企业全栈蓝图”**。

![](https://mmbiz.qpic.cn/mmbiz_jpg/iaM6L8b5B1GmHkBicPzsxjawS27J0rlJx6oIwUIia38xpkA8IYTOmAWbjw75fs1OIAD6DBhP6IKmeHBcBSqFjg31qZ9LGXdOdD8DvNNV2L9Zhk/640?wx_fmt=jpeg&from=appmsg)

这是一个由五层架构严密咬合的生态系统：

1.  **AI Hypercomputer（底层算力）：** 专为智能体物理特性优化的极速底座；
    
2.  **Agentic Data Cloud（数据引擎）：** 破解暗数据，提供零移动的“可信业务语境”；
    
3.  **Agentic Defense（安全防御）：** 以机器速度运行的自主免疫系统；
    
4.  **Agentic Platform & Models（中枢平台与模型）：** 包含前沿模型与全生命周期治理的任务控制中心；
    
5.  **Agentic Taskforce（数字特遣队）：** 蓄势待发、重塑客户体验与内部生产力的专用智能体。
    

看似是五个产品，但 Google 这次更想让企业接受的，是不同以往的全新买法，不是买几个零件回去自己拼，而是直接买一套企业智能体操作系统。芯片是底座，模型是内核，数据是上下文，Gemini Enterprise Agent Platform 负责统一调度。

意思很直白：不要再试图用不同厂商的开源模型、拼凑的向量数据库和第三方的监控工具去搭建“草台班子”了，我能给你一套能**在全域环境下丝滑运转的工业级解决方案。**

![](https://mmbiz.qpic.cn/mmbiz_png/VeJKXItpwPUzKbub0MRKx61l9dsOUESzKb3kWktSQzuVHts4BVMpbCYA4ibs3rVyec3cY0TYTJtCa2Axw9X5g1Q/640?wx_fmt=png)

### **为智能体时代准备的算力“杀手锏” ：Google 第八代 TPU**

为了支撑起这套庞大的操作系统，Google 首先在物理世界进行了一场极致的自我革命。

过去几年，业界一直试图用一种通用的大算力芯片打天下。但 Google Cloud AI 基础设施首席技术官 **Amin Vahdat** 带来了一个截然不同的物理学洞察：步入智能体时代，训练（Training）和推理（Inference）的需求已经彻底分化。

训练模型，要的是吞吐量、带宽、集群规模。真正把大量智能体跑进生产环境里，企业更在乎的是延迟、并发和成本。拿训练芯片去承接高频交互，本质上不划算，就像拿重卡去送外卖，能送，但不合适。

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/iaM6L8b5B1GmPGoxxaFueyhg57sukYagxAP4Icoh7SdIPY1PsUkGG4Xu5iau30cnEKYPib8x5GGmmr6PsOPUuic6BWIvsqwf3rIZ25zPId9wL7E/640?wx_fmt=jpeg&from=appmsg)

基于这一判断，Google 史无前例地推出了拆分架构的第八代 TPU（TPU v8），将底层算力供给生生劈成了两半：

一半是**代号为 Sunfish 的 TPU 8t**。这头由 Broadcom 协助设计的“训练怪兽”，通过将块级乘法直接移入 MXU 内部实现原生量化，单个 Pod 提供 121 Exaflops 的 FP4 算力，配备了令人咋舌的 2 PB 共享高带宽内存——足以装下**美国国会图书馆全部数字藏品的 100 遍**。结合高达 47 PB/s 无阻塞带宽的全新 Virgo 网络，Google 现在能将百万个 TPU 融为一体，将极其庞大的前沿模型训练周期从数月暴力压缩至几周。

另一半，则是**代号为 Zebrafish 的 TPU 8i**。这才是 Google 专为智能体时代准备的“杀手锏”。由 MediaTek 协助设计的 8i 完全为低延迟推理优化，通过将内存缓存完全托管在芯片上，打破了阻碍长上下文解码的“内存墙”，将延迟暴降了 5 倍。凭借全新的蝴蝶拓扑结构，它能在单 Pod 中以近乎零的延迟运行数百万个并发的智能体。

再往下看，Google 还给高交互场景配了 C4N 和 M4N 实例，也在继续推自研 Arm 架构的 Axion CPU。总之，这么多年 TPU 的路线其实都很相似：**算力不再由单一芯片定义，而是整个数据中心要跟着任务类型去重新组织。**  

像 Citadel Securities 这种对延迟极端敏感、每天处理超过 5000 亿美元交易的客户，迁到这套设施后，速度提升 2 到 4 倍，成本还能降 30%。这类案例，也是 Google 想证明系统重构有价值的地方。

![](https://mmbiz.qpic.cn/mmbiz_png/VeJKXItpwPUzKbub0MRKx61l9dsOUESzckpUw8tJc4LcXmaJLmFzIhPH3LwHjFZicHp78td0MU1x83kQXFmAcIw/640?wx_fmt=png)

### **跨越“幻觉”死局：数据层才是真正的卡点与护城河**

光有模型和算力，还远远不够。企业 AI 一旦真往业务里走，最难的地方很快就会冒出来，数据不在一处，上下文也不完整。

Google Cloud 首席产品与业务官 **Karthik Narain** 在台上说得很直接：没有上下文的推理，本质上就是盲猜。

很多企业今天并不缺模型，缺的是可信的业务语境。真正有价值的数据，散在 PDF、会议录音、SaaS 应用和不同云厂商的数据库里。模型再聪明，拿不到这些上下文，也很难做出稳定判断。

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/iaM6L8b5B1Gnux9J9JdgIO50OnaXep1WtYy7a53kwS2QqVfcvN7cPzPfbt76vhXMqMB4jbuc8NOpicQbp9TArpLFevUy2ZUs2GRx1rfDkI0OY/640?wx_fmt=jpeg&from=appmsg)

Google 这次重构的 Agentic Data Cloud，基本就是冲着这个卡点去的。里面最值得看的，有两个东西。

第一个是 **Knowledge Catalog。**

过去企业想让 AI 理解 PDF、图片、文档，前面往往要先做一大堆数据清洗、向量化、图谱构建。Google 现在想把这件事往前收。按照它的演示，一份非结构化图片或 PDF 进入 Google Cloud 之后，Smart Storage 会先自动打标签，Knowledge Catalog 再继续往下读文件、抽实体、连关系。

大会上那个“午夜漩涡冻酸奶”的演示就很典型。新配方里看不出大豆过敏原，普通关键词搜索也搜不到。但系统顺着 Knowledge Catalog 去翻暗数据，最后在另一份原本没被关联起来的供应商手册 PDF 里，找到了一种名叫 Base 204 的原料，并确认里面含有大豆。原来可能要查好几周的事，被压到了几秒钟。

第二个是 **Cross-Cloud Lakehouse**。

这一步其实很关键。过去云厂商都想让企业把数据搬到自己的云上，Google 这次承认了另一件事，大企业的数据天然就是多云、分散的。所以它基于 Apache Iceberg，允许智能体直接对 AWS S3 和 Microsoft Azure 里的数据做低延迟推理，而且不需要先把数据整批搬过来。

再加上 Lightning Engine for Spark，Google 讲的是另一种能力，智能体可以直接跨云找数据、做匹配、跑模拟。这对 Google 来说也不只是补能力，它是在借智能体这件事，把自己的系统往别人的云里接。

![](https://mmbiz.qpic.cn/mmbiz_png/VeJKXItpwPUzKbub0MRKx61l9dsOUESzKm2eAVLxGTw8BNeRibxo35vVooAwvM5bGfplhAkeuwWRQkBolzmluZA/640?wx_fmt=png)

### **没有刹车，智能体就是脱缰野马**

企业真正担心的，从来不只是 AI 能干什么，还有它会不会把事情搞砸。

如果成千上万个拿到执行权限的 Agent 可以随意调 API、查数据库、发财务邮件，那等于把一批不完全可控的数字员工直接放进公司系统里。这个问题不解决，规模化落地就很难往前走。

Google Cloud 首席运营官兼安全产品总裁 **Francis deSouza** 这次提到一个词，**影子 AI**。意思是，那些已经在企业内部跑起来，却没被统一看见、统一管理的智能体。

所以 Google 这次把治理和安全放得很靠前。在平台治理层，它推出了 **Agent Identity（智能体身份）**和 **Agent Gateway（零信任网关）**。在 Gemini Enterprise 平台里，每个智能体都可以拿到唯一加密 ID 和授权策略，每一次调用、每一步跨库查询，都能通过 Agent Observability 做可视化记录，再配合 Model Armor 做隔离。

这套东西说白了，就是给数字员工补一套身份体系、访问边界和监控能力。

而在主动安全防御层面，**Google 此前花费 320 亿美元天价收购云安全巨头 Wiz，这次终于是伏笔回收了。**  

![](https://mmbiz.qpic.cn/sz_mmbiz_png/iaM6L8b5B1GlicpIrmvWwSekMMP9AY23IVPiasbiaqeDYKyozufLsiaBWG1Cexw39KI9J0ITKNerZQunz5JqIUaia1icQzawOuNiadvibGYZE7E6iabmk/640?wx_fmt=png&from=appmsg)

面对从初始访问到移交黑客组织的交接时间已经锐减到 22 秒的现代攻击，人类分析师已经彻底出局。Wiz 联合创始人 Yinon Costica 展示了由 Wiz 和 Google SecOps 联合打造的 **Agentic SecOps（智能体安全运营中心）**。

这套系统部署了专家级的 AI 智能体特遣队：

**红色智能体（Red Agent）** 就像不知疲倦的白帽黑客，从外部不断探测企业云环境中暴露的智能体接口（例如演示中发现的身份验证绕过漏洞）；

**绿色智能体（Green Agent）** 会自动将漏洞进行风险分类，精准定位到导致风险的那行代码，并将修复 PR（拉取请求）直接发送给开发者的编程 IDE 中；

**蓝色智能体（Blue Agent）** 则结合 Google 的全球威胁遥测，将原本需要 30 分钟的调查时间极限压缩到 60 秒。

这是以机器的速度对抗机器。Google 想证明的是，安全不应该永远是事后补丁，而要在系统里提前长出来。

![](https://mmbiz.qpic.cn/mmbiz_png/VeJKXItpwPUzKbub0MRKx61l9dsOUESz48jReib7YT0TxNT4t33TpLpxQj2uQ9EGt8roialmVuXpDqwqbxNqeyOA/640?wx_fmt=png)

### **从 Demo 走向“数字劳动力”的流水线**

把算力、数据和安全底座夯实之后，真正的舞台交给了 **Gemini Enterprise Agent Platform**。

这是企业智能体时代名副其实的“任务控制中心”。

这次模型层的更新很多。不仅有专为跨域规划与自主执行而生的 **Gemini 3.1 Pro**，更有专攻高吞吐量多媒体处理的 **Veo 3.1 Lite（视频）**、**Lyria 3 Pro（音频）** 以及 **Nano Banana 2（极速高保真视觉）**。此外，架构极具创新性、采用混合注意力机制（Hybrid Attention）的 **Gemma 4**，也极大地拉低了端侧和边缘侧执行代理任务的门槛。

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/iaM6L8b5B1GmicmIlAMgDWaic4QhBxBkyTdTnr5zMfVMBtotQg6ibNIcoTjbo5Z01SBJB46HULNUR0R27Q3y2iaZMZZbtJNpEJGJTNXNdc43AOcg/640?wx_fmt=jpeg&from=appmsg)

但比模型列表更重要的，是平台层怎么把这些东西装进企业流程里。

比如 **Low-Code Agent Studio**，Google 想把做 Agent 这件事，从工程团队专属能力，往业务团队手里再推一步。没有编程基础的人，也可以用自然语言把企业内部逻辑接到大模型上。

再比如 **MCP，Model Context Protocol**。Google 这次对 MCP 的支持很高调。GCP 自己的服务、Atlassian、ServiceNow，甚至客户自建数据库，都可以通过 MCP 变成智能体可调用的能力。

这类发布背后其实都是一个方向，Google 不想让 Agent 永远停在 demo 阶段。它想把 Agent 变成可以进企业、受治理、能交接、能汇报的数字劳动力。

![](https://mmbiz.qpic.cn/mmbiz_png/VeJKXItpwPUzKbub0MRKx61l9dsOUESznR9XfkE5uI0iapqUCKXAtVOhMbPU9F4yUYYH7Ivd39vN2xT6ABBjVlQ/640?wx_fmt=png)

### **Agentic 时代，开发者的定义正在变宽**

在 Google Cloud Next 2026 的现场，我们还邂逅了 Google Cloud Developer Marketing Senior Director（开发者营销高级总监）**Ricky Robinett**。

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/iaM6L8b5B1GniaqhcAHPPUmibat6NOAq8t1q2E6us0iaJvgpACfeAx00rgGuChFDNgvdarGSbCQ54LUialEYiceADwHV72xiaNLzPICic5DGPkicXr0c/640?wx_fmt=jpeg)

和主舞台上那些关于模型、平台与 Agent 的宏大叙事相比，和他的这段交流把 Google 这次真正想讲给开发者听的东西落得更实了一层。

如今的变化不只是今天可以更快地写出多少代码，或者多调用几个模型 API，而是**开发者面对系统的方式本身正在变化。**  

过去做传统应用时，很多问题还可以靠经验、脚本和一串 console.log 往下排查，但一旦开始构建真正会自主规划、自主调用工具、跨系统执行任务的智能体应用，开发者很快就会发现，老办法不再够用，系统需要的是一整套围绕可观测性、编排、治理和平台能力重建起来的新基础设施。

更有意思的变化还在另一层。Ricky 反复提到，随着 Agentic Coding 持续降低门槛，未来真正稀缺的能力，可能不再只是掌握多少语言、框架和工程细节，而是能不能**像开发者一样思考，知道如何定义问题、拆解流程、组织上下文，再把一个模糊的想法变成真正可运行的产品。**

也正因为如此，“开发者”这个群体本身也在被重新定义。过去它更多指向传统工程师，但在这一轮变化里，产品经理、运营者，甚至原本不在技术路径里的人，都可能因为自然语言、低代码工具和 Agentic 平台的成熟，第一次真正进入构建应用的流程。

![](https://mmbiz.qpic.cn/mmbiz_png/VeJKXItpwPUzKbub0MRKx61l9dsOUESzsibRzBZe5iaXXeg6Z0HCjGjywO4xytRq2j57yv1du1kAvwoxDwbyWXXw/640?wx_fmt=png)

### **争夺终极入口：Workspace 不仅是工具，更是操作台**

但对 Google 来说，把开发者一侧的平台层讲清楚，还不是终点。

再强大的后台引擎，如果员工不用，也只是一堆昂贵的代码。所以 Google 这次还有一个很明显的动作，继续抢**工作入口**。它盯的不是单独某个 AI 助手按钮，而是员工每天真正待得最久的那层界面。

Google Workspace 产品副总裁 **Yulie Kwon Kim** 在台上举了一个很常见的白领场景，你只是想回答一个简单问题，结果十分钟之后已经打开了十几个标签页，一半时间在找信息，一半时间在想怎么处理。

Google 这次发布的 **Workspace Intelligence**，想解决的就是这个问题。它不是 Gmail 里多一个功能，也不是 Docs 里多一个助手，而是一个横在 Gmail、Docs、Slides、Chat 上面的统一智能层。

![](https://mmbiz.qpic.cn/sz_mmbiz_png/iaM6L8b5B1GkjOl3GFnElaI4JficCM0icWHumBxGQjBsnzQnsicCpEpKPnpqjibFxkYmGY9Fav1d7GGKic4zqBXKfNTjhNaZUcHgEP5xoBcAXpPT4/640?wx_fmt=png&from=appmsg)

在区域经销商筹备发布会的演示中，当用户在 Google Chat 收到一条新品到货通知时，Workspace Intelligence 充当了全能的调度员。它能听懂“找出上个季度那份带有区域销售图表的推介手册”这样极为口语化且依赖历史记忆的指令，瞬间在海量云盘中揪出目标；紧接着，它能调用 HubSpot 的实时盈亏数据，应用企业的品牌视觉规范，直接在后台自主生成一份完整的 Google Slides 汇报材料。

与此同时，在更为专业的设计与开发领域，极具科幻感的 **Vibe Design** 理念与 **Google Stitch** 工具，更是将这种“意图直达结果”的入口体验推向了极致。员工只需通过语音和无限画布，描述想要的“氛围”或业务逻辑，系统就能直接生成高保真 UI 和机器可读的 Design.md 规范代码。

你给出语音、画布和业务逻辑，系统去生成高保真 UI 和可读代码规范。这也是同一条路，**把人的意图尽量直接接到结果上。**

![](https://mmbiz.qpic.cn/mmbiz_png/VeJKXItpwPUzKbub0MRKx61l9dsOUESzqx2xW7plD6avyia60VVbQKliaEzOicfg2jkMnn8ckF4F5M19SYZzdxp6w/640?wx_fmt=png)

### **开放，是刺向“围墙花园”最锋利的剑**

将整套系统打包售卖，最容易引发企业的恐惧：我会不会被 Google 彻底锁定？

有意思的是，Google 这次反而非常强调开放。它不仅支持 Anthropic 的模型，包括最新的 **Claude Opus 4.7**，也在底层接入了 NVIDIA Vera Rubin NVL72，还通过 Cross-Cloud Lakehouse 去打通 AWS 和 Azure 的数据，并继续增强和 Microsoft 365 的互操作性。

结合其豪掷 7.5 亿美元支持埃森哲、德勤等十几万生态合作伙伴的“生态基金”，**Google 其实看得很清楚，大企业天然害怕被单一厂商锁死。**

当别人想用封闭的 API 和大模型把你关进围墙花园，Google 这次放开给你自由的硬件选择、无界的数据直连和全球顶尖的模型底座（无论是他们的还是对手的）。但只要你想要获得最好的治理、最强劲的网络编排和最丝滑的生态融合——你最终还是会回到 Google 打造的这套底层架构上。

### **独家解读：云计算服务模式正在被系统性重写**

AI 已经不再是屏幕上那些讨巧的文字对话框，它正在以计算的速度，解析并重构物理世界与商业世界的每一寸肌理。

过去咱们每逢发模型就“爆”个不停，爆了三年之久，现在押注某个“ChatGPT 时刻”的年代也该过去了。接下来的十年，科技巨头之间比拼的，将是谁能把庞杂的算力、无序的数据、严苛的治理、全自主的安全以及数以亿计的员工入口，天衣无缝地焊成一套**能够自我进化的企业级运转系统**。

以下是 CSDN 高级副总裁、奇点智能研究院院长李建忠的独家解读：

**1\. Google 正在重构云计算的底层范式**

Google Cloud Next 2026 最值得关注的，并不是某一个模型或产品，而是**云计算服务模式正在被系统性重写**。

过去二十年，云计算的分层结构 IaaS/PaaS/SaaS 是清晰而稳定的。但在智能体成为执行主体之后，这一分层逻辑正在发生结构性坍塌。Google 此次提出的 Unified Stack，本质上并不是产品组合的升级，而是一次**对云计算分层逻辑的重构**。

在**基础设施层**，Google 通过第八代 TPU 与 AI Hypercomputer 架构，将训练与推理分离，IaaS的本质已经从提供“硬件资源租赁”转向提供“围绕 Token 生产的智能输出能力”。

在**平台层**，PaaS 正在被拆解并重组为两类核心能力的组合：一类是以 Gemini 为代表的 MaaS（Model as a Service），负责提供标准化的模型能力供给；另一类则是以 Gemini Enterprise Agent Platform 为核心的 Agent Runtime，负责智能体的执行、调度与治理。这一变化意味着，云平台不再只是“部署应用”，而是在运行“具备自主决策能力的执行单元”。

在**应用层**，SaaS 正在从“功能交付”走向“任务交付”。传统软件要求用户在系统中逐步操作以完成任务，而在 Agent 驱动的服务中，用户只需表达意图，多智能体协同以结果为中心进行交付，SaaS 被重构为 AaaS（Agentic as a Service）。

**2\. 企业数字劳动力的“操作系统”**

如果说云计算的底层结构正在被重写，那么这次发布中最关键的“中枢变量”，就是 Gemini Enterprise Agent Platform。它试图将 Agent 从“零散的实验组件”，升级为“可管理、可治理、可规模化运行的智能体操作系统”。

这个平台并不是孤立存在的，而是嵌入在 Google Cloud 整个 Unified Stack 之中：底层由 TPU 与 AI Hypercomputer 提供算力，中间通过 Agentic Data Cloud 提供业务语境，Agentic Defense 提供安全防御，Agentic Platform & Models，提供模型与智能体全生命周期治理，上层通过 Workspace Intelligence 接入员工日常工作入口。

如果说过去云计算解决的是“计算资源的供给和应用问题”，那么在智能体时代，Google 试图解决的，是“数字劳动力如何被组织、调度与治理”的问题。
