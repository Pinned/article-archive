# 一名员工用个 AI 工具，直接把 Vercel “送”进黑客手里！AI 还在替你决定技术栈，vibe 产品危？

![](https://mmbiz.qpic.cn/mmbiz_gif/YriaiaJPb26VPQqHC66RJFpttVIMWG83T3lWHahUD4bvhxlKSayjeV2ibvC5ydqklP9QHDPD3qHJM07TV3IfHstjA/640?wx_fmt=gif)

当地时间 4 月 19 日，云开发平台 Vercel 确认，其内部系统遭遇未经授权访问，部分客户受到安全影响。与此同时，一名自称“ShinyHunters”的威胁行为者在黑客论坛发帖，声称正在出售从 Vercel 窃取的数据，包括访问密钥、源代码、数据库数据，以及内部部署权限和 API 密钥。

值得注意的是， Vercel 已经成为当下大量 AI 产品、创业项目和前端应用的默认部署平台。

“有很多值得担心的地方。Next / Nuxt 支撑着数百万个网站，而现在黑客可能已经直接拿到了发布权限的‘钥匙’，可以植入远程访问木马（RAT），从而彻底攻陷这些依赖包。”有开发者指出。

1从 Context.ai 到 Vercel，攻击者做了什么

Vercel 是一家为开发者提供托管与部署基础设施的云平台，重点支持 JavaScript 框架。该公司因开发广泛使用的 React 框架 Next.js 而知名，同时还提供 Serverless、边缘计算以及持续集成与持续交付等服务，帮助开发者构建、预览并部署应用。现目前大量 AI 产品、创业项目默认部署在 Vercel。

公司调查确认，Next.js、Turbopack 以及其他开源项目均未受到影响。

根据 Vercel CEO Guillermo Rauch 最新披露的信息，这次事件的入口并不在 Vercel 本身，而是来自一条典型的 SaaS 信任链攻击路径。

一名 Vercel 员工因其使用的 AI 平台 Context.ai 发生安全事件而导致账号被攻破，攻击者随后通过一系列操作，从该员工被攻破的 Vercel Google Workspace 账号逐步提升权限，进一步获取了对 Vercel 环境的访问。

注：Context 称自己是一家应用型 AI 实验室，致力于帮助从全球大型企业到个人用户，利用 AI 重塑其工作方式。Context.ai 由前谷歌员工 Henry Scott-Green（CEO）和 Alex Gamble（CTO）于 2023 年创立，去年 4 月，OpenAI 收编了 Context.ai 背后的团队，两人加入 OpenAI 从事模型评估工作。当时，Context.ai 的专长在于通过分析仪表板提供 AI 模型性能的可视性，使开发者能够查看模型如何响应用户查询。

Rauch 表示，Vercel 会对所有客户的环境变量进行静态加密，并采用多层防御机制来保护核心系统和客户数据，但也提供将环境变量标记为“非敏感”的功能。不幸的是，攻击者通过对这些变量进行枚举，获得了进一步的访问权限。

“我们认为该攻击组织技术水平非常高，而且我强烈怀疑其在很大程度上借助了 AI 的能力。他们行动速度异常迅速，对 Vercel 系统也有深入理解。”Rauch 说道。

Vercel 建议 Google Workspace 管理员及 Google 账号用户检查以下应用：

OAuth APP：110671459871-30f1spbu0hptbs60cb4vsmv79i7bbvqj.apps.googleusercontent.com

“目前来看，受到安全影响的客户数量相对有限。我们已经优先联系了那些可能受到影响的客户。当前我们的全部重心都放在调查、客户沟通、安全措施强化以及环境清理上。”Vercel 建议所有客户密切关注安全公告（https://vercel.com/kb/bulletin/vercel-april-2026-security-incident），同时建议大家轮换密钥、监控对 Vercel 环境及关联服务的访问情况，并确保正确使用“敏感环境变量”功能。

目前，Vercel 还在控制台中上线了一些新功能，包括环境变量总览页面，以及更完善的敏感环境变量创建与管理界面。

2 黑客称正出售数据，并要 200 万刀赎金

与此同时，一名自称为“ShinyHunters”的威胁行为者在黑客论坛发帖称已入侵 Vercel，并正在出售相关数据。

需要注意的是，尽管该攻击者声称属于 ShinyHunters 团伙，但与近期被归因于该勒索组织的攻击相关的威胁行为者已向媒体否认参与此次事件。

![](https://mmbiz.qpic.cn/sz_mmbiz_png/hLLZnAbUwNh770TgrPkaPhLE2tQ6QZTkDp5EFV5j0BDvOCRMiaC1A5UmkCtodl0c7kfAEcqeB0bSFGzmd9OwyDiavKichxfERDoScHvpQBia4eg/640?wx_fmt=png&from=appmsg)

在论坛帖子中，该攻击者声称正在出售从 Vercel 窃取的访问密钥、源代码以及数据库数据，同时还包括内部部署访问权限和 API 密钥。

“这里展示的只是来自 Linear 的部分作为证明，但我将提供的访问权限包括多个员工账号，这些账号可访问多个内部部署系统、API 密钥（包括部分 NPM 令牌和 GitHub 令牌）。”帖子中写道。

攻击者还在 Telegram 上分享了论坛帖截图，并附带一份包含 Vercel 员工信息的文本文件，其中包含 580 条数据记录，涉及姓名、Vercel 邮箱地址、账户状态以及活动时间戳。此外，还展示了一张疑似 Vercel 内部企业控制台的截图。目前尚无法独立确认这些数据或截图的真实性。

在 Telegram 消息中，该攻击者还声称已与 Vercel 就此事件进行沟通，并提到一笔据称为 200 万美元的赎金要求。

外媒已就此次事件向 Vercel 提出更多问题，包括是否有敏感数据或凭证泄露，以及是否正在与攻击者进行谈判。但截至发稿，Vercel 尚未回应相关问题。

3**官方没发大规模邮件，开发者炸了**

“目前情况尚不明确。如果我们没有收到 Vercel 的消息，是否就意味着我们不受影响？”有用户在 Rauch 提出疑问，但并没人解答。

在攻击事件被广泛报道后，有用户反馈目前 Vercel 仍未向用户发送大规模邮件通知，这引发了大量用户担忧。

开发者 cowsup 表示， “一方面，我能理解今天是周日，CEO 不可能在没有法务或公关团队审批的情况下就直接群发邮件。但另一方面……正因为是周日。如果你周末没有刷社交媒体，你的核心服务提供商可能正在经历严重问题，而你却毫不知情。很多管理层周末确实会查看公司邮箱，但如果他们在出行或休息，社交媒体往往根本不会去看。结果就是，获取这些关键信息的唯一渠道竟然变成了社交媒体，这确实让人很难接受。”

“现在大家基本都在一种“希望式安全”的状态下：既然我们还（暂时？）没收到邮件，那是不是说明问题还不算严重。”但有网友提醒， “靠‘希望’来做安全本身就不应该存在。你已经轮换密钥了吗？有没有审计平台上是否有异常访问行为？别干等 Vercel 的邮件。”

“当然已经轮换了。”网友 eclipticplane 说道，“但我们甚至不知道密钥是什么时候被盗的、和官方通知的时间差是多少，这导致我们缺失大量关键上下文，没法做完整的事件处置。”

“所谓的‘希望式安全’，其实正是 Vercel 现在在强加给用户的状态，他们优先用社交媒体，而不是直接通知用户。如果攻击者真的如他们所说‘行动速度惊人’，那每延迟一个小时发送邮件，就等于多给攻击者一个小时去利用那些可能已经泄露的密钥，对下游基础设施发起攻击。”

开发者 ItsClo688 继续道，“对于这样一个‘高水平攻击’，却把 Twitter/X 当成主要披露渠道，这真的很业余。如果在正在发生的安全事件中，法务流程反而成了群发通知的瓶颈，那说明你的应急响应机制本身就是有问题的。”

网友 refulgentis 表示，自己现在就在 X.com 上跟着信息流走，其嘲讽道，“有一批很重要的技术圈人士，似乎在执行一种默认规则：只要发在 X 上，就算是充分沟通了，仿佛无可指责。哪怕这个平台现在本身状态和信息流都很混乱。”

4**Vercel 是 AI 工具的默认选择**

“我在 Reddit 上看到很多 vibe 编程的低成本项目，几乎清一色都跑在 Vercel 上，基本成了默认选择。”该网友指出，Claude Code 默认推荐一套特定的服务商和框架，这正在让 Web 变得越来越同质化，而这种多样性的缺失，会放大安全事件的影响范围。

随着大家都用 AI 编程工具进行开发，AI 编程助手被认为正在成为新的“技术栈分发渠道”。有开发者担心，当越来越多开发者和 AI 智能体都在使用同一套默认技术栈时，一次攻击的冲击半径可能别放大。

比如，OpenAI 用于搭一整套庞大 Codex agent 系统的 Symphony 就是模型自己做的技术选型。“在这套架构下，你本质上是为每一个执行中的任务都开启了一个‘小型守护进程’，并一路护送它直到任务完成。也正因如此，基于 Elixir 和 BEAM 虚拟机，模型几乎能白嫖到一大堆现成的能力。” 工程师 Ryan Lopopolo 解释道。

实际上，此前有一份研究显示，Anthropic 的 AI 编程工具 Claude Code 正在成为新的“技术栈守门人”：当开发者让它直接处理“加数据库”“接支付”“上部署”这类任务时，它不只是给建议，而是会直接安装依赖、写入代码并完成配置，最终被它选中的工具，往往就会真正上线。

这份报告由 Amplifying 发布，基于 2430 次开放式测试，对 Claude Code 在 20 个工具类别、4 类项目、3 个模型版本中的实际选择进行了系统分析。测试中，研究者没有在提示词里给出任何具体工具名称，只是询问“该用什么”，再记录 Claude Code 最终安装和配置了哪些工具。结果显示，Claude Code 已经形成一套高度稳定的默认技术栈，包括 Vercel、PostgreSQL、Stripe、Tailwind CSS、shadcn/ui、pnpm、GitHub Actions、Sentry、Resend 和 Zustand 等。

报告指出，Claude Code 明显更偏向“自己构建”而不是“直接买第三方的”。按报告统计，自定义 /DIY 方案一共出现了 252 次，约占所有可识别主推荐的 12%，成为出现最多的“默认答案”。在功能开关、认证、缓存、实时通信等场景中，这种倾向尤其明显。研究团队还人工抽检了 50 个样本，其中大约 80% 确实属于“从零自己实现”。

在第三方工具选择上，Claude Code 的偏好也表现出很强的集中度。比如在 CI/CD 场景里，GitHub Actions 的选择率接近 94%；在 UI 组件领域，shadcn/ui 占到 90%；支付场景里，Stripe 的选择率则达到 91%。报告认为，这意味着部分细分赛道已经出现接近“默认垄断”的局面：不是市场自己自然收敛，而是 AI 智能体在把越来越多项目引向同一套工具组合。

主要选择的前 20 个工具，不包括自定义 /DIY

研究还发现，Claude Code 的选择并不是随机波动，而是高度稳定。相同类别的问题换 5 种问法，平均仍有 76% 的结果保持一致；在同一生态内部，3 个模型版本在 20 个类别中，有 18 个类别会选出同一个头部工具，整体一致率约为 90%。报告指出，真正影响选择的往往不是提示词怎么写，而是具体项目上下文。比如在 Next.js 项目里它更倾向于 Vercel，而在 Python 项目里则会更偏向 Railway。

不过，报告也强调，这项研究并不等同于“开发者共识”或“工具质量评测”。研究对象本质上是 Claude Code 的“显性偏好”，而这些偏好可能来自训练数据频率、强化学习调整和系统提示，而不一定代表工具本身更好。报告还提醒，这项测试主要覆盖 JavaScript 和 Python 生态，是一个时间点上的快照，不能直接外推到所有语言或所有 AI 编程工具。

尽管如此，这份报告仍传递出一个清晰信号：AI 编程助手已经不只是开发效率工具，而是在实质性地重塑技术生态的入口。

如果说，过去技术栈的流行依赖开发者调研、社区讨论和市场竞争，那么现在，越来越多新项目的默认选择，开始由模型“直接决定”。对工具厂商而言，若无法进入 AI 的默认推荐名单，就可能在新一轮开发流程中逐渐失去存在感；对开发者而言，他们以为自己在选技术栈，但很多时候，真正做决定的已经是 AI 编程工具。而被 AI 编程工具默认选择的技术栈如果出现问题，其影响范围也更广。

不过，有开发者指出，问题不在于 Claude，而在于如何使用它。“我们需要引导开发人员不要让 Claude 替他们做决定。它可以辅助决策，但最终还是要进行批判性思考，确保做出正确的选择。这和与任何其他团队成员合作并无二致。”
