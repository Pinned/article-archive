# 18年老用户“哭着离开”：因GitHub频繁宕机，Anthropic官方推荐、5.2万Star项目Ghostty官宣迁出！

![](https://mmbiz.qpic.cn/mmbiz_gif/Pn4Sm0RsAujqPrtpN5QqqOuSG0AdUUx8wRg3vvr7aQJicdC0cHDBqZrJBZajQUXx97BwNaAE6L6x7OGlsTIpWjw/640?wx_fmt=gif)

整理 | 郑丽媛  

出品 | CSDN（ID：CSDNnews）

当一个用了 18 年、几乎“住在 GitHub 上”的开发者，突然宣布要离开，原因会是什么呢？

最近，HashiCorp 联合创始人、Terraform 和 Vagrant 的作者 Mitchell Hashimoto 在个人博客中写下了一篇带着明显情绪的长文，核心只有一句话：GitHub 近来故障太多、太不稳定了。

而他的解决方案也很直接——把自己当前最重要、拥有 5.2 万 Star 的开源项目 Ghostty，迁出 GitHub。

![](https://mmbiz.qpic.cn/mmbiz_png/S1iaf4GgGjEyterichcyicxM3B4FwJ5nV0riboEHWfCNibEFOw0oHRNKibXJ50zRwXrkIg4UPkSD8KuhPcal6x5l4RrYficHNLvX6FTZAbldQV4aL8/640?wx_fmt=png&from=appmsg)  

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06QuwjkSP1G6wEJHaJCLTONqlcQexqRgJcIICxofIOJs6B6tWBfibb7now/640?wx_fmt=png)

一个老用户的告别：从热爱到失望

先简单介绍一下，Ghostty 是一款用 Zig 语言编写的终端模拟器，其设计目标是在 macOS 和 Linux 上提供快速、功能丰富、平台原生的终端体验，未来还计划支持 Windows。值得一提的是，连 Anthropic 官方都曾推荐 Ghostty 作为 Claude Code 的首选终端。

![](https://mmbiz.qpic.cn/sz_mmbiz_png/S1iaf4GgGjEzesPTVvOuysII4UW3ueSkJwH5YOJ6bIz0pxRDZeK4YchAE8hJDFEvLMOE12B42ibMrle92M6MKQhic4FpQbVePiaVDCTibL8SVs1s/640?wx_fmt=png&from=appmsg)

本周，Ghostty 的项目作者 Hashimoto 在其个人博客中写道，他是 GitHub 的第 1299 位用户，注册时间可以追溯到 2008 年 2 月。从那之后的 18 年里，他几乎每天都会打开 GitHub，而且一天不止一次。

对他来说，GitHub 几乎占据了自己的半个人生。Hashimoto 回忆称，过去无论经历什么阶段，GitHub 都是他最愿意待的地方：

● 大学时期熬夜写代码，临睡前还要再提交一次 commit；

● 度假时会收藏一堆项目仓库链接，研究别人如何组织开源协作流程；

● 甚至连蜜月旅行期间，也会趁清晨空档打开 GitHub 看看动态。

在他眼里，GitHub 从来不只是代码托管平台，更像互联网世界里的精神家园。Hashimoto还自曝，自己早年开发 Vagrant 的重要动力之一，就是希望靠这个项目拿到 GitHub 的工作机会。他在 20 岁时曾公开打趣道：“如果这个项目做得够好，也许 GitHub 会录用我。”

虽然最终没有加入 GitHub，但 Hashimoto 始终把那里视为理想中的开发圣地。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06QDyP5HTUOXsPlJWd79yygiasaqXicpN7ibIfqiak5WFpaxE1mGxxpfmMjiaA/640?wx_fmt=png)

一个月日志：几乎“天天宕机”的开发体验

让 Hashimoto 真正感到失望并决定离开的，并不是某一次事故，而是 GitHub 持续性的“不稳定”。

为了避免主观情绪干扰判断，他做了一件很工程师的事情：记录数据。在过去一个月里，只要 GitHub 出现问题、影响到他的工作，就在当天日期旁边打一个“X”。

结果是：几乎每天都有 X。

Hashimoto 强调，这些问题并不只是偶发卡顿，而会直接影响核心开发流程：

● GitHub Actions 无法运行，CI/CD 中断 

● Pull Request 页面卡死，代码审核无法推进 

● 页面加载异常，影响日常操作 

甚至在他写博客当天，就因为 GitHub Actions 故障，连续约两小时无法进行 PR 审核。

对于普通用户来说，这也许只是一次短暂异常；但对于依赖 GitHub 完成开发、测试、发布全流程的项目来说，这种级别的中断，已经不是“体验不好”，而是工作被硬生生中断。

于是，Hashimoto 给出的结论也非常直接：如果一个平台每天都让你被卡住几个小时，它就已经不再适合正常工作了。他在博客中写下了这样一段话：

> “我想待在这里，但它不想让我待在这里。
> 
> 我想把工作做完，但它不想让我完成工作。
> 
> 我想发布软件，但它不想让我发布软件。”

对于一个长期依赖 GitHub 开发、协作、发布项目的人来说，这已经不是抱怨，而是信任崩塌。Hashimoto 甚至承认，最近自己对 GitHub 的愤怒和批评带有明显的私人情绪——因为他“过度爱上了一个产品”，所以失望情绪也格外强烈。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06Q0QtgZAx8xsoTReptvArfwbn9MvHGVfV98Qkl5PMRS6MCt2ljwLJwoQ/640?wx_fmt=png)

Ghostty 成为“第一个离开的项目”

在具体行动上，Hashimoto 并没有“一刀切”地抛弃 GitHub：Ghostty 将逐步迁移到新平台；GitHub 上保留只读镜像仓库，方便现有用户继续访问代码；他的个人项目将暂时继续留在 GitHub。

换句话说，这更像是一次“核心业务迁移”，而不是彻底告别。

之所以优先迁移 Ghostty，原因也很现实：Ghostty 是他本人、维护者团队以及开源社区受 GitHub 故障影响最明显的项目，因此必须优先处理。

至于下一站去哪，Hashimoto 坦言目前仍未最终确定，团队正在与多个候选平台沟通，其中既有商业服务商，也有自由开源方案（FOSS）。  

此外，由于该项目当前已深度依赖 GitHub 的诸多功能，迁移不会一夜完成，而会以渐进方式推进。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06Q3SAo8dibFicWvibnB5u4gdHLgb2AbA6UJ8VImKnyW92hibqZefwpDbPYAQ/640?wx_fmt=png)

所以，GitHub 发生了什么？

过去，GitHub 作为“开发者基础设施”的地位几乎不可撼动，无论是代码托管、协作流程，还是 CI/CD 体系，都高度集中在这一平台上。

但近期，稳定性问题确实开始频繁出现：GitHub Actions 多次中断、PR 流程异常、API 响应不稳定、页面加载失败……例如就在 4 月底，一次由 Elasticsearch 配置问题引发的事故，导致大量 Pull Request 无法正常完成。

这些问题单看并不罕见，但当频率上升到“影响日常工作”的程度时，问题的性质就变了。

更微妙的是，众所周知 GitHub 近年来正在大力推进 AI 能力，例如 Copilot、自动化代码分析等。与此同时，微软整体战略也在向 AI 倾斜。为此，不少开发者开始提出一个疑问：

当平台资源不断向 AI 功能倾斜时，GitHub 的基础稳定性是否被忽视了？

虽然，目前并没有直接证据证明两者存在因果关系，但至少在时间线上，两者是重叠的。Hashimoto 本人也并没有明确指责这一点，但他的经历，确实发生在这一阶段。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06Q0jqAkXB2hEohZkOqYzXVsmJtnNRQncYh54ZQOpcC5ZGSicYFgtBZP4w/640?wx_fmt=png)

如果 GitHub 变好了，他还是愿意回来

最终，Hashimoto 在博客中留下了一段颇为沉重的告别：“我希望 GitHub 变得更好，但我也想写代码。现在我已经没法继续依赖 GitHub 写代码了。抱歉，18 年后，我得走了。”

在 Hacker News 的后续回应中，他还透露，自己写下这篇博客时甚至哭了，眼泪一度滴到了键盘上：“说来也怪，谁会因为一个 SaaS 平台而哭呢？但 GitHub 对我来说远不止于此，我和它之间有着一种不健康的依恋。它给予了我太多，我也对此心怀感激。但，它已不再是曾经的模样。”

Hashimoto 说，直到真正写下文章并点击发布按钮的那一刻，这场离别才变得真实。

不过，正如上文所说，这次迁出 Ghostty 并非意味着彻底决裂——Hashimoto 表示，如果未来 GitHub 能拿出真正的改进成果，而不是停留在承诺和口号层面，他还是愿意回来。
