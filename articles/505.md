# 对外卖Copilot，对内用Claude？微软内部正在悄悄换“编程大脑”

![](https://mmbiz.qpic.cn/mmbiz_gif/Pn4Sm0RsAujqPrtpN5QqqOuSG0AdUUx8wRg3vvr7aQJicdC0cHDBqZrJBZajQUXx97BwNaAE6L6x7OGlsTIpWjw/640?wx_fmt=gif)

整理 | 郑丽媛  

出品 | CSDN（ID：CSDNnews）

如果你最近在关注 AI 编程工具，可能会注意到一个微妙变化：Claude Code 的名字，出现得越来越频繁了。

而这一次，并不是某个初创公司在“蹭热度”，而是——微软自己，正在大规模把 Claude Code 引入内部。

要知道，微软对外销售的“官方 AI 编程工具”，一直是 GitHub Copilot。但如今在微软内部，从核心工程团队到非技术岗位，Claude Code 的使用范围正在迅速扩大，甚至已经覆盖到 Windows、Office、Edge 等关键产品线相关部门。

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06QuwjkSP1G6wEJHaJCLTONqlcQexqRgJcIICxofIOJs6B6tWBfibb7now/640?wx_fmt=png)

从模型试用，到工具级全面引入

微软与 Anthropic 的关系，其实并不是最近才开始的。

早在 2025 年 6 月，微软就已经在其开发者部门内部，引入了 Anthropic 的 Claude Sonnet 4 模型；随后，在 GitHub Copilot 的部分付费用户中，也逐渐增加了 Claude 模型的使用权重。

不过，那时还只停留在“模型层”。

真正的变化发生在最近：微软不再只是用 Claude 的模型，而是开始直接推广 Anthropic 的完整产品——Claude Code。也就是说，Claude 不再只是 Copilot 背后的一个“可替换大模型选项”，而是以独立工具的身份，进入微软工程体系。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06QDyP5HTUOXsPlJWd79yygiasaqXicpN7ibIfqiak5WFpaxE1mGxxpfmMjiaA/640?wx_fmt=png)

Windows、Office、Teams 背后的团队，都在试用 Claude Code

目前，微软新成立的 AI 工程组织 CoreAI 团队，已经对 Claude Code 进行了数月测试。该团队由前 Meta 工程负责人 Jay Parikh 领导，被视为微软未来 AI 工程能力的中枢。

更关键的是，微软并没有把 Claude Code 限定在“工程师专属工具”里。

就在不久前，微软 Experiences + Devices 事业部（负责 Windows、Microsoft 365、Outlook、Teams、Bing、Edge、Surface 等核心产品）开始被统一要求安装 Claude Code。

同时，微软还在内部明确鼓励：即便没有任何编程背景的员工，也可以尝试使用 Claude Code。例如设计师、产品经理、项目负责人，微软允许、甚至期待他们能用 AI 直接生成原型代码、验证想法、提交改动。

据知情人士透露，如今微软的软件工程师被要求同时使用 Claude Code 和 GitHub Copilot，并对两者进行对比反馈。

这就形成了一个有些“微妙”的局面：

● 对外：微软仍将 GitHub Copilot 作为首选 AI 编程工具出售给客户。

● 对内：如果这些大规模内部试点效果理想，不排除未来微软会将 Claude Code 直接卖给云客户。

不过，可以看出微软并不急于给出“标准答案”，而是在认真评估 Copilot 是否为最优解。

事实上，从开发者社区的反馈来看，相较于其他AI编程助手，Claude Code 的优势并不在“代码生成速度”，而在于：上手门槛低、多轮对话中上下文理解稳定、对非技术用户更友好、更适合做“探索式编码”和原型验证。

这些特点，正好契合了微软当前的一个重要目标：让更多岗位参与到“代码生产”中来。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06Q0QtgZAx8xsoTReptvArfwbn9MvHGVfV98Qkl5PMRS6MCt2ljwLJwoQ/640?wx_fmt=png)

拉近与 Anthropic 的距离，对 OpenAI “变心”？

而根据外媒《The Information》的报道，微软已成为 Anthropic 最大的客户之一。更反常的是，微软甚至允许其销售团队将 Anthropic 模型的销售额计入 Azure 的业绩指标——而过去，这种待遇微软通常只留给自研产品或 OpenAI。

在 2025 年 11 月，微软与 Anthropic 还签署了一项重量级协议：Microsoft Foundry 客户可以访问包括 Claude Sonnet 4.5、Claude Opus 4.1、Claude Haiku 4.5 在内的一系列模型。

而作为交易的一部分，Anthropic 也承诺采购高达 300 亿美元的 Azure 计算资源。

与此同时，微软近期还开始在 Microsoft 365 应用和 Copilot 的部分功能中，优先使用 Anthropic 的 Claude 模型——尤其是在那些 Anthropic 表现明显优于 OpenAI 模型的场景下。

为此，外界频频质疑：那 OpenAI 呢，微软真的“变心”了吗？

微软传播负责人 Frank Shaw回应道：“企业定期测试和试用竞争产品，是为了更好地理解市场格局。OpenAI 还是我们在前沿模型上的主要合作伙伴，我们也将继续长期合作。”

这句话并不假，但也没有否认另一件事：微软已不再把鸡蛋放在同一个篮子里了。

  

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06Q3SAo8dibFicWvibnB5u4gdHLgb2AbA6UJ8VImKnyW92hibqZefwpDbPYAQ/640?wx_fmt=png)

10 万个代码仓库，会发生什么变化？

说到这里，真正值得关注的问题或许是：

Claude Code 在微软内部使用规模的扩大，意味着什么？

微软曾在去年披露，其 91% 的工程团队已经在使用 GitHub Copilot，AI 工具主要用于加速一些重复、枯燥的工程任务。但随着 Claude Code 和 Claude Cowork 的引入，Anthropic 正在通过 AI Agent 能力，让编程和非编程任务都变得更加“有参与性”。

显然，微软应该看中了 Claude Code 的低门槛特性，希望让更多非技术员工也能借助 AI 直接提交代码。可同时，它也进一步加剧了业内对初级程序员岗位正在被 AI 挤压的担忧。

那么，你平时会使用 AI 编程助手吗，最常使用的又是哪个呢？
