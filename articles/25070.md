# openJiuwen 开源 Symphony 技能编排与分发系统，支持海量技能的精准发现、稳定协同

![](https://mmbiz.qpic.cn/mmbiz_gif/Pn4Sm0RsAujqPrtpN5QqqOuSG0AdUUx8wRg3vvr7aQJicdC0cHDBqZrJBZajQUXx97BwNaAE6L6x7OGlsTIpWjw/640?wx_fmt=gif&wxfrom=5&wx_lazy=1&tp=webp#imgIndex=0)

当一个 Agent 能调用的 Skill 从 5 个膨胀到 50 个，会发生什么？

答案是：任务执行效果出现"断崖式下滑"——技能越多，Agent 反而越选不准。

这是当下 Agent 工程绕不开的难题：如何正确选择、调度和执行各类 AI 可执行单元（如Agent、Skill、MCP 工具）？

业界最新动向是，华为支持的 **openJiuwen** 社区刚刚开源了一套 Skills 编排与智能分发系统——**Jiuwen Symphony**，实现对技能的动态选择与编排协作。

据了解，这套系统主打两件事：让 Agent 在海量技能里"又快又准"地挑出能用的技能，再把候选技能按依赖关系图自动编排成可执行轨迹。

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06QuwjkSP1G6wEJHaJCLTONqlcQexqRgJcIICxofIOJs6B6tWBfibb7now/640?wx_fmt=png&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=3)

Jiuwen Symphony 想解决什么问题？
------------------------

不管是出行规划、还是金融投资，或是办公自动化等复杂任务场景，技能生态绕不开两个问题：**怎么选**和**怎么用**。

*   **选不准**：技能太多，上下文窗口装不下所有技能说明书，必然出现信息过载、上下文过长的问题。若改用语义检索查找相似技能，由于检索与任务执行相互独立，仍无法解决多意图、隐式意图这类需要推理的选择问题。
    
*   用不好：技能选对了，还需要精准编排。技能广场上的技能标准不统一，上一个技能的产出能否喂给下一个、谁先谁后、谁依赖谁，这些关系若每次让模型临场发挥，常常出现执行失败、流程中断等稳定性问题。
    

针对这两个问题，Jiuwen Symphony 给出一套技能分发的解决方案：通过统一能力表征，**动态检索递归技能树、智能编排技能依赖图**，实现 Skill 的全周期精细化管理——让海量技能随用户需求，奏出一支完整的交响乐(Symphony)。

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06QDyP5HTUOXsPlJWd79yygiasaqXicpN7ibIfqiak5WFpaxE1mGxxpfmMjiaA/640?wx_fmt=png&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=4)

双核心架构：检索用树，编排用图
---------------

Jiuwen Symphony 由两个核心组件构成：**技能检索**与**技能编排**。前者负责从海量技能里选对结果，解决选不准的问题；后者负责把合适的技能按依赖串成可执行轨迹，解决用不好的问题。

### 技能检索：让 LLM 自己"递归作曲"，实现层次化的技能树检索

将用户本地或技能广场中的 skill 预先组织成一棵层次化技能树，原本平铺的技能列表就变成了可逐层浏览的技能目录。

检索时，LLM 不必一次性读完全部技能，而是在树上**按任务需求逐步导航**：先判断主能力分支，再顺着输入格式、输出目标、验证要求等线索展开相关节点，直到定位候选技能。

这种方式既保留了 LLM 对复杂语义任务的判断力，又降低了无关技能对上下文和注意力的干扰。

![](https://mmbiz.qpic.cn/mmbiz_png/S1iaf4GgGjEzdHSKE7ianreNNjUdCXlt7alV7sd7VZXU5luqms3qKmle66QXMfToevhG4kia0z8ADUia3KxMbotulpzdEpqZneKetUS8GAd357A/640?wx_fmt=png&from=appmsg)

更进一步，这套检索机制并没有被设计成执行前的一次性预处理，而是被整合为一组面向 Agent 的专用工具，例如：

*   递归能力树构建
    
*   分支探索
    
*   轻量预览
    
*   候选技能读取
    

这意味着，**技能选择被真正融入了 Agent loop，而非检索与执行相互独立**。Agent 在理解任务、尝试求解、发现信息不足或判断需要专门能力时，可以随时动态调用检索工具，按需查看技能目录、读取候选技能说明，再决定下一步执行路径。

这种上下文感知的设计带来了三个直接收益：

*   检索行为贴近当前推理上下文，选择更符合真实任务状态；
    

*   技能调用可以随着任务推进动态调整，而不是一开始“拍脑袋选完”；
    
*   避免全量注入 skills 带来的上下文膨胀，让大规模技能库仍然保持可用性。
    

换句话说，Jiuwen Symphony 不是在做一个的技能在线搜索器，而是在为 Agent 提供一种**面向执行过程的动态能力分发机制**。

效果如何？团队给出了实测数据。下图是该功能在 **SkillsBench** 下的表现：

![](https://mmbiz.qpic.cn/sz_mmbiz_png/S1iaf4GgGjEwQ11doZw3pSTwZb7vMFrA5Lqa1uUjmkB9YHhRBecBPEdEPuolYI5jZLBLOMb34MPaA5uVbQ3PVEX9Ck8EwxPvueibE7pP55Jt4/640?wx_fmt=png&from=appmsg)

Symphony 技能检索在 Top-5 选择上取得最佳效果：召回率 0.741、成功率 0.615，均高于关闭检索和 embedding 两条基线；同时把平均最大上下文从约 82k tokens 降到约 21.5k tokens。在大规模技能场景下，按需探索技能目录能更准确、也更省上下文地找到可用技能。

### 技能编排：把技能关系谱成依赖图，用图编排出执行轨迹

如果说技能检索解决的是"选谁上场"，那么技能编排要回答的就是"怎么让它们配合起来"——

围绕任务目标，对技能的输入输出、衔接关系和调用顺序做结构化编排。

编排系统先离线给技能提取能力指纹，形成统一表征。指纹覆盖技能的基础信息、输入输出产物和能力质量评估。再借助动态词表，把命名不一的同义字段（如 body、content、正文）归一到规范词上。这样，技能不再是一个个孤立的调用对象，而是变成**可比较、可计算、可关联的能力 schema**。

Jiuwen Symphony 基于能力指纹构建技能关系图，并作为结构化资产沉淀下来。图里的节点是技能，边可以表示为技能A的输出能否成为技能 B 的输入。如此一来，**在 Agent 执行任务时关系图谱可以被反复读取、复用和演进**。

![](https://mmbiz.qpic.cn/sz_mmbiz_png/S1iaf4GgGjEyoB2pqh1qE9SIXPP5Uw19FzFWu5XUd2DTOXicXOJPnX6q8exDlCywdfYuiblowbFHEiaYMC7icMuoFvBNek0xDUXM15eUWeR4aSBM/640?wx_fmt=png&from=appmsg)

运行时，编排不是简单地从列表里挑一个工具，而是先依据用户目标、已有输入和候选技能锁定一组种子技能，再沿着总谱里的可衔接关系做双向搜索：向后扩展"下一步做什么"，遇到缺失输入就反向回溯"谁能补上"，从而拼出真正接得上的技能组合。

这种方式带来三个直接收益：

*   组合可靠
    

不再只凭"看起来相关"，而是建立在输入输出能否真正衔接之上，中间缺失的步骤还能被自动补齐；

*   协同可解释
    

每个技能为何被选中、上一步产物如何进入下一步、当前缺什么输入，都能从执行轨迹里清晰追溯；

*   失败可局部修复
    

某个技能执行失败时，可沿图结构定位，围绕该节点就近替换或修补，而不必整条链路推倒重跑。

至此，技能的选择与组合不再依赖每次临场发挥，而是被升级为一种"基于图谱的系统化组织"——这也是 Symphony 区别于一般工具调用机制的核心点。

![](https://mmbiz.qpic.cn/mmbiz_png/Pn4Sm0RsAujX5kS5KQ6BaBUsy1RqR06Q0QtgZAx8xsoTReptvArfwbn9MvHGVfV98Qkl5PMRS6MCt2ljwLJwoQ/640?wx_fmt=png&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=8)

场景一：视频处理（仅使用技能检索）

当用户安装的技能从几十个增长到数千甚至更多之后，真正困难的地方不再是“有没有技能”，而是 Agent 能不能在合适的时候找到正确技能，并把它们组合进当前任务。Symphony 技能检索就是为这个场景设计的。任务执行时，Agent 不再一次性面对庞大的技能列表，而是像查目录一样，按任务需要逐步探索相关分支，找到真正需要的技能。

![](https://mmbiz.qpic.cn/mmbiz_png/S1iaf4GgGjEweYicGef5EkqtDDTrVTCRu4e4BPM7swRAqq5pY2vAj54aORibptb4RPEdnC1Vau5icchWz5t8IJLicSllaPibVDkEVIFZp69z4avOc/640?wx_fmt=png&from=appmsg)

以一个视频处理任务为例。用户只需要输入自然语言需求：

![](https://mmbiz.qpic.cn/mmbiz_png/S1iaf4GgGjEyyCNR8rYZeQYcXrKHuPyQonXETW5uYMzWiaDjRRHHfwJibpDtB2iaD2FLO8sKRDGs1eiaRboFJbiaf5me7wcKecMgooRx2eQLmRpgE/640?wx_fmt=png&from=appmsg)

Agent 会先理解这个任务所需的多类能力，并通过技能目录进行逐步检索。它不是直接从上千个技能里硬找，而是先探索“视频/音频处理”这类主能力分支，再根据任务目标继续查看“文本整理”“报告生成”等辅助分支。

![](https://mmbiz.qpic.cn/sz_mmbiz_png/S1iaf4GgGjEyEGKlmk0W7s2gicOP7J1YXQFKY7zZic6icTmrRbWeXAyRF9vMEiap8jPYRaBMibzpmdTHoXx7QhMibg0AWRaicnK3gzicdF0vCXsnbibdY/640?wx_fmt=png&from=appmsg)

在这个过程中，Agent 会逐步发现任务所需的关键技能。例如，视频处理技能用于读取和剪辑素材，音频转写技能用于生成文本。

![](https://mmbiz.qpic.cn/mmbiz_png/S1iaf4GgGjExQMHTF5jTU1cWlIXwWFbIWIEToeibJnLGuDJXlsVh8RjzH1od4ws20T3HIOZUNrMAy0JiaWxugPKHYOzEkDL2tnHYEDESmvvGuE/640?wx_fmt=png&from=appmsg)

最终，Agent 得到的不是一个孤立技能，而是覆盖了任务所需的各个领域的技能：

*   视频读取与处理
    
*   音频转写
    
*   口头禅检测
    
*   停顿分析
    
*   关键片段整理
    
*   报告生成
    
*   备选/兜底处理
    

![](https://mmbiz.qpic.cn/sz_mmbiz_png/S1iaf4GgGjEyUjlTRdZpxpyPwOOLcaSaz4qeswiaeITicmUWQvv11hPVR0WeqWd1cXxicYoVqfoqY8q9Y0cua6sR6G4SPxmydwDdzeiaeCGiadEH4/640?wx_fmt=png&from=appmsg)

这个过程体现了 Symphony 技能检索的核心价值：当技能规模变大时，系统不再依赖模型一次性记住所有技能，而是把技能变成可导航、可探索、可按需披露的目录。Agent 在执行任务的过程中主动查找技能、理解技能、组合技能，从而让大规模技能库真正变成可用的能力池。

### 场景二：办公写作（ 仅使用技能编排）

以一个办公写作任务为例。用户只需要输入自然语言需求：把英文博客截图（介绍Skills相关内容）翻译后整理成公众号文案，并发送到指定邮箱。

![](https://mmbiz.qpic.cn/sz_mmbiz_png/S1iaf4GgGjEzVetwdicqvyoulAiatHjibic8AH5bDIwGsf2M2UJMWBltd5LzfooeGoCqKLAzsacjQzvzkYe3J9j6baKenZlzKH6JWbHf5QicrEHrk/640?wx_fmt=png&from=appmsg)

如果没有技能编排，Agent 可能会先调用图片翻译技能，但输出仍然是一张翻译后的图片。这个结果对用户看图有用，却无法直接交给后续写作技能继续改写。接下来系统可能转去读取博客网页，或者重新猜测原文来源，最终产出就会偏离用户真正提供的材料。

![](https://mmbiz.qpic.cn/sz_mmbiz_png/S1iaf4GgGjExloRxX1JS2Hf6ibewbicwicibAJcM1JKDynkfdXzU2Xh6B5dyQZevtnEAkShA5ald0WhtbcMx4rqvLNaDKQxHEfMt9ZSIiafS5jlEA/640?wx_fmt=png&from=appmsg)

有了技能编排后，系统会先围绕任务目标检查每个技能的输入输出关系。它会发现：后续的公众号文案撰写需要可编辑文本，而不是图片；邮件发送需要正文内容和收件人，而不是截图文件。

![](https://mmbiz.qpic.cn/mmbiz_png/S1iaf4GgGjEyogjnW8whRU18BXZZy5Qjnr4icphD2FxGRAE8T4yicG0K4iaxBcXnQHClXm9uk4AKO3lJurWyXKZ8EyicgzicSPVG6k9rlz7OJPzC8/640?wx_fmt=png&from=appmsg)

因此，系统会更稳定地规划出一条可衔接的路径：

*   图片翻译
    
*   文字识别
    
*   文案撰写
    
*   邮件发送
    

![](https://mmbiz.qpic.cn/mmbiz_png/S1iaf4GgGjEyoKX2FxUnia2BHr7kbUFVrAo5lyNANLPE9qrEFicr6SGXqibx2HWaJn8SBD5kI4RibBpJz8tf98iap7vVgOUmDS2AClCRZRjlYSBdY/640?wx_fmt=png&from=appmsg)

  

在这条技能链里，图片翻译负责让截图内容变成用户可理解的信息，文字识别负责把图片中的内容转成可编辑文本，文案撰写技能再基于文本生成公众号风格的内容，最后邮件发送技能把整理好的正文发送出去。

这个过程体现了 Symphony 技能编排的核心价值：系统不只是知道“有哪些技能”，还会判断技能之间是否真的能接上。只有当前一步的输出能够满足下一步的输入要求时，这条连接才会被保留下来。这样，多个原本独立的技能才能被组织成一条稳定的工作流，避免任务在中途变成图片、网页、文本之间的反复猜测，最终让复杂办公任务真正形成结果闭环。

结语
--

今天，越来越多团队在构建 Agent 时都会遇到同一个现实问题：模型能力在增强，但系统能力没有同步增长。

*   模型可以理解复杂需求，却常常无法稳定调用外部能力；
    
*   工具越来越多，却难以被高质量地检索和复用；
    
*   流程越来越复杂，却缺乏统一的技能组织方式。
    

而openJiuwen这次的探索，提供了一个很值得关注的方向：**把 skill 当作需要被管理、检索、调度、编排的“系统资产”**，而不只是提示词里附带的一段工具说明。

它的价值不止于“提升一次任务成功率”，更在于为 Agent 的工程化扩展提供了一种可持续路径：

*   当技能库继续扩大时，系统仍能保持**可检索性**
    
*   当任务链条继续拉长时，系统仍能保持**可组织性**
    
*   当执行环境不断变化时，系统仍能保持**动态适配能力**
    

从这个意义上说，openJiuwen 提出的 Jiuwen Symphony 并不是一个局部优化技巧，而是一种面向下一代 Agent 基础设施的思路：**让技能这类AI可执行单元不再只是被调用，而是被系统化地“演奏”出来**。

通过“技能检索 + 技能编排”的双核心架构，这套系统把海量 skills 从静态资源转化为**动态可调用、可组织、可扩展的能力网络**，也为复杂任务执行中的精细化 skill 管理提供了新的可能。

当 Agent 进入真正的大规模技能时代，如何让能力**被准确发现、被高效调度、被稳定协同**，也许会成为比模型参数更重要的竞争力。

而这场关于“技能交响”的探索，或许才刚刚开始。
