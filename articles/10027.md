# Cursor 3 发布：IDE 不重要了，智能体控制台上位，VS Code 这一套开始失效

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/hLLZnAbUwNiax9B9dicdyKHDxTxPWewqGzIPBGok7erA4InsibyVy2I3icpcj4Bb4RDDn15grMaDwbfoaVH1hKO0qlMZMKjcb8KfQrCj42BfV3E/640?wx_fmt=jpeg&from=appmsg)

  Cursor 3 用智能体管理控制台取代了传统代码编辑器，标志着 AI 辅助开发工具与开发者工作流程均已发生重大转变。

作为同类产品中营收增长最快的 AI 代码编辑器，Cursor 发布了首款非代码编辑器产品。

Cursor 3（代号 Glass）从零开始构建而成，将智能体管理控制台作为主界面，首次将传统 IDE 置于次要位置。工程师们仍然可以在其中编写代码，但这款产品的核心设计理念已经转为“用户会将大部分时间用于调度智能体、审查输出及决定发布哪些任务”。如今，原本显示文件树的位置已经被提示词输入框所取代。

![](https://mmbiz.qpic.cn/sz_mmbiz_png/hLLZnAbUwNia0rpOHyIcqS7eX4J3gCBLtmNYGTvtxHUFw3M9P3yGZw69wibImnZ1kcCyeJabKwDU8qWO2ibaFYfATyaA5BBianloJlm2Zbb4Ajk/640?wx_fmt=png&from=appmsg)

几个月来，编码工具的地位正被 AI 模型甩得越来越远。而 Cursor 3 的出现，则是这家年收入 20 亿美元的公司接受这一现实、并将全部产品线押注于此的直接结果。这种转变与基础设施工程师们的过往经历如出一辙。

云控制面板取代了 SSH 终端。Kubernetes 控制器取代了手动服务器配置。如今，编排层正替换掉代码编辑器成为主要界面。尽管趋势相同，但这波变革的风险更主同，因为被降级的抽象层乃是开发者们使用长达 40 年的底层架构。

Cursor 3 带来了什么

Cursor 于 2022 年作为 VS Code 的一个 fork 诞生，如今则已经衍生出完全围绕智能体构建的兄弟版本。Cursor 方面表示，新界面“完全从零开始构建，以智能体为中心”，并将传统 IDE 视为可随时切换的备选方案。

工作区默认支持多仓库，智能体和用户可以同时在不同仓库中操作，而所有本地和云端智能体都会显示在统一的侧边栏中。该侧边栏会从 Cursor 的所有界面中提取智能体，包括从移动设备、Web 客户端、Slack、GitHub 乃至 Linear 启动的会话。云智能体会生成演示与工作截图，这样工程师们在查看输出时无需将代码拉取至本地，即可快速了解变更内容。

新版本中最突出的功能当数 Cloud Handoff（云交接）。大家可以将正在运行的智能体会话从笔记本电脑移动至 Cursor 云端，确保其在关闭计算机时继续运作，并在准备就绪后拉回本地在桌面上进行编辑和测试，反之亦然。用户可以在云端启动任务，并在需要接管控制时将其撤回。这种在本地和云环境之间进行会话移植的能力，一直是多数同类竞品的最大短板。

大家可以将 Cursor 3 的出现，视为从管理单一服务器向着通过控制平面进行服务器集群管理的重大转变。虽然用户仍可在必要时通过 SSH 接入服务器，但控制平面已经成为决策制定、工作负载分配与查看系统状态的枢纽。Cursor 3 对智能体的处理方式同样迎来了重大转变，其中 IDE 相当于 SSH，Glass 则相当于控制平面。

转型源自压力

Cursor 3 的发布并非偶然。过去六周以来，Cursor 公司展开了一场我称之为“加速产品攻势”的行动。3 月下旬，《财富》杂志发表了一篇专题报道，将 Cursor 的处境描述为“创新者困局”的典型案例，并很快引发强烈反响。此前，彭博社刚刚指出 Cursor 的年化收入在 2026 年 2 月正式突破 20 亿美元，在三个月内就翻了一番。（延伸阅读：《[Cursor 经历生死存亡](https://mp.weixin.qq.com/s?__biz=MjM5MDE0Mjc4MA==&mid=2651277616&idx=1&sn=93d33fabd5541e797fb5640a07ef0496&scene=21#wechat_redirect)》。）

但《财富》杂志发现，Anthropic 公司打造的终端优先编程智能体 Claude Code 仅用一年多就将年化收入拉升至 25 亿美元，赢得超过 30 万家企业客户的青睐。开发者们纷纷公开表示要脱离 Cursor、转投 Claude Code 怀抱。一位 Cursor 投资人则告诉《财富》杂志，他投资组合中的多家初创公司正考虑与 Cursor 解约。3 月份有报道称，尽管 Cursor 正在寻求新一轮融资、估值也来到约 500 亿美元，但市场关于其后续发展态势的信心正在瓦解。

为此，Cursor 迅速采取三项重大举措以应对这些挑战。3 月 5 日，该公司发布 Automations 自动化系统，无需人工干预即可根据 GutHub 事件、Slack 消息与定时器来触发智能体。

3 月 19 日，Cursor 又发布了 Composer 2，这是其首个基于月之暗面开源 Kimi K2.5 打造的自主研发模型。

Cursor 声称，Composer 2 在其专有 CursorBench 测试中得分达 61.3，高于 Claude Opus 4.6 的 58.2 分，且单元 token 成本更低。需要注意的是，这里的 CursorBench 是 Cursor 自家打造的评估套件。

同样在 3 月，Cursor 启用了自托管云智能体，允许财富五百强企业在其内部基础设施上运行 Cursor 智能体。

随后就是 Cursor 3 的正式发布。一个月内三款产品，界面也经历了全面重构。这种节奏，正是该公司坚信所处业务领域正在经历重新定义的明证。

结构性转变

Cursor 转型的重要意义，在于提示出一种远超单一企业的范式化变革。如今，所有 AI 辅助开发领域的主要参与者均已达成共识，即智能体需要独属于自己的编排界面。各厂商的分歧在于这套界面应该位于何处，并由此构成了开发者工具架构中颇为有趣的差异。

Anthropic 的 Claude Code 采用终端优先模式，即彻底放弃集成开发环境（IDE）。命令行界面（CLI）就是编排层，由开发者在 shell 中通过自然语言命令进行工作。Anthropic 随后在 claude.ai/code 中添加了基于浏览器的界面外加桌面应用程序，但仍将终端视为核心，也就是让编排层完全独立于编辑器之外。

OpenAI 则选择了另外一条道路。Codex 如今涵盖独立桌面应用程序、命令行界面、面向 VS Code 及其分叉的 IDE 扩展，加上位于 chatgpt.com/codex 的云界面。其中，桌面版应用程序成为管理并行智能体、查看差异以及跨项目运行工作的“指挥中心”。OpenAI 的理念是，编排层应当无处不在，开发者无论使用哪种界面都可随时访问。

> “过去四十年间，软件开发范式由代码编辑器定义。Cursor 3 则豪掷一切，坚信监督智能体的重要性必将超过编辑文件。”

谷歌的方案与 Cursor 最为接近。在向 Windsurf 支付了 24 亿美元授权费，并将其 CEO 与核心工程师招入 DeepMind 之后，谷歌推出了 Antigravity——这是一款以智能体为中心的 IDE，且提供两种截然不同的模式。编辑器视图包含传统编程环境，管理器界面则允许开发者在工作风内并行生成、编排及观察多个智能体。智能体编排与代码编辑共存于同一应用程序之内，但分别由不同视图负责呈现。

Cursor 3 也采用了这种内部 IDE 模式，只是进一步做出了强化。Glass 将智能体控制台设为默认视图，而编辑器则成为备选视图。谷歌 Antigravity 认为这两种视图同等重要，体现的是侧重点上的差异。换言之，两家公司对于开发者最终会将时间花在哪些方面有着不同的判断。

我们可以将当初云基础设施的演进与这种分歧做个类比。AWS 推出管理控制台时，用户并不会停止使用 SSH。但 SSH 不再是制定基础设施决策的地方，控制台开始成为控制平面，SSH 则成为偶尔用到的调试工具。

Anthropic 和 OpenAI 认为编排层可以作为独立的应用程序保持运转，与 IDE 分离开来。Cursor 和谷歌则认为编排层需要与编辑器共存，这样在智能体出错时，开发者就仍能直接查看代码。业界普遍认为智能体编排将成为新的主界面，只是对具体架构设计尚未达成共识。

这对开发者们意味着什么？

目前，这场变革的实际影响主要体现在以下三个方面：

1\. 模型选择成为基础设施

Cursor 将 Composer 2 设为默认模型，但仍允许开发者根据对话情况切换至 Claude、GPT-5.4 和 Gemini。驱动智能体的模型现在成为基础设施层面的决策，类似于选择数据库或云区域，而 token 经济效应会随着规模扩大而逐渐形成。

Cursor 公布的 Composer 2 标准版定价为每百万输入 token 0.50 美元，每百万输出 token 2.50 美元（截至 2026 年 3 月的价格），远低于 Anthropic 及 OpenAI 的前沿模型定价。

对于运行数十个并行智能体的团队而言，这样的定价无疑比功能设置或 UI 偏好更能吸引开发者们的目光。

2\. VS Code 的护城河正在干涸

Cursor 从 VS Code 分叉而来，在继承其扩展生态系统之余实现了即时分发。而 Cursor 3 则另辟蹊径，旨在打造差异化优势。如果以智能体为中心的界面设计最终胜出，VS Code 扩展的重要性将一路降低。微软显然需要密切关注这一趋势。

VS Code 作为开发者工具中心的假设已经维持了近十年，但如今正在逐渐瓦解。

JetBrains 也面临着同样的压力。随着主要交互界面从编辑文件转向智能体管理，传统 IDE 在编程智能与重构工具方面的竞争优势也将烟消云散。

3\. 工作流程首先变化，开发职位随后转型

Cursor 3 认为，工程师们会耗费大量时间来审查智能体生成的差异，验证云智能体生成的截屏截图，决定哪些任务要推送至云端、哪些任务保留在本地，以及管理 PR 工作流程。

这与以往的代码编写方式与技能需求截然不同，意味着编程开始更多偏向工程经理或者平台运维岗位，而非传统软件开发者。换言之，软件工程师的角色正在向应用层系统运维者的角色融合。

> “软件工程师的角色正在向应用层系统运维者的角色融合。”

Cursor 自身的发展轨迹也体现了这种转变。该公司于 2025 年 12 月收购了代码审查平台 Graphite，正如公司 CEO Michael Truell 所言，随着 AI 推动代码生成的加快，代码审查正成为新的瓶颈。

Cursor 3 将这一逻辑进一步延伸，即由智能体编写代码、Graphite 负责审查，而工程师的工作则是对二者加以协调。集成开发环境（IDE）的重要性则退居其次。

未来展望

AI 编程智能体的编排层如今已经成为新的产品类别。所有主流厂商都已推出了相关产品，目前悬而未决的问题在于架构层面。编排层应当存在于 IDE 之内、之外，还是同时存在于所有界面？

Anthropic 和 OpenAI 押注于独立工具形态，Cursor 与谷歌则押注于集成有 IDE 的综合控制台。谁的选择与真实趋势更贴合，谁就能在未来十年间赢得开发者们的拥护，正如 2010 年代初的云控制平面之争决定了当下的基础设施巨头地位一样。

在过去四十年间，软件的构建方式一直由代码编辑器所定义。Cursor 3 押注于这样一种理念：监督智能体要比编辑文件更重要。Cursor 并不是要扼杀 IDE，只是降低了 IDE 的地位。

如果此判断准确无误，那么出现在我们面前的很可能就是人类记忆中的最后一款代码编辑器了。未来如何，让我们拭目以待。
